* Introduction

Featurizer is a PostgreSQL-based implementation of Deep Feature Synthesis (DFS) for temporal datasources.  It generates pure SQL queries that materialize features across entity-relationship graphs, where child tables represent event logs --- sparse, irregularly-timestamped records such as inspections, transactions, clinical visits, or complaints.

The system follows a three-stage pipeline architecture:

1. *Planner* --- performs DFS traversal of the entity graph, synthesizes features through aggregations and transformations, and produces intermediate CTE definitions.
2. *SQL Renderer* --- converts planner output into a final PostgreSQL query using CTEs wrapped in a =CROSS JOIN LATERAL= against =as_of_dates=.
3. *Executor* --- optional database execution layer that runs queries and returns pandas DataFrames.

This document provides a systematic gap analysis: what time-series and event-data feature families the featurizer currently implements, which families are feasible to add within the SQL + event data paradigm, which are fundamentally incompatible, and which sit in a borderline zone.  The goal is to guide the project roadmap by separating engineering effort from theoretical impossibility.

The constraints that bound this analysis are twofold.  First, all feature computation must be expressible as a single PostgreSQL query (CTEs, window functions, lateral joins, subqueries --- but no procedural extensions, no PL/pgSQL loops, no external UDFs).  Second, the data model is entity-relationship graphs where child tables are event logs with irregular timestamps, not dense equispaced time series.

* What We Mean by Event Data

** Events vs. Dense Time Series

The distinction between event data and dense time series is fundamental to understanding which feature families apply.

*Event data* consists of discrete records generated by real-world actions: a patient visits a clinic, a ship docks at a port, an inspector files a report, a customer places an order.  These records share several properties:

- *Irregular timestamps* --- events happen when they happen.  A patient may visit twice in one week and then not again for six months.  There is no fixed sampling rate.
- *Meaningful gaps* --- the absence of an event is itself informative.  No inspections for 90 days means something different from daily inspections.  Gaps are not missing data; they are signal.
- *Categorical + temporal signal* --- the "value" of an event is often a category (type of inspection, diagnosis code, complaint category) combined with its timestamp, not a continuous measurement.
- *Entity-scoped* --- events belong to entities (a customer, a facility, a vessel).  Features are computed per-entity, partitioned by the entity's primary key.

*Dense time series* consists of measurements taken at regular intervals: sensor readings every second, stock prices every minute, daily temperature recordings.  These series have:

- *Fixed sampling frequency* --- the time between observations is constant or nearly so.
- *Continuous values* --- each observation is a real-valued measurement.
- *No meaningful gaps* --- a missing reading is an error to be imputed, not a signal.
- *Stationarity assumptions* --- many classical techniques (ARIMA, spectral analysis) assume or require stationarity after differencing.

* Feasible Extensions

The following feature families can be implemented within the existing SQL-only constraint and are well-suited for event data.  They are ordered by implementation priority, from trivial extensions of existing infrastructure to architecturally novel additions.

** P1: Additional Percentiles and Dispersion Measures

*** Description

Extend the existing =OrderedSetAggregator= pattern to produce a family of percentile-based features: p10, p25, p75, p90, p95, p99, interquartile range (IQR), and coefficient of variation (CV).

*** Why It Matters for Event Data

Percentiles reveal the shape of event distributions better than mean/stddev, which assume symmetry.  For event data --- where distributions are often heavily skewed (many small transactions, few large ones) --- percentile features capture tail behavior that drives ML model performance.  IQR provides a robust spread measure unaffected by extreme events.  CV (stddev/mean) normalizes variability across entities with different activity levels.

*** SQL Implementation

#+NAME: src:percentile-aggregation
#+begin_src sql :exports code
-- p25 and p75 via ordered-set aggregate
select
    entity_id,
    percentile_cont(0.25) within group (order by amount) as p25_amount,
    percentile_cont(0.75) within group (order by amount) as p75_amount,
    percentile_cont(0.75) within group (order by amount)
      - percentile_cont(0.25) within group (order by amount) as iqr_amount,
    -- Coefficient of variation
    case when avg(amount) = 0 then null
         else stddev(amount) / avg(amount)
    end as cv_amount
from events
group by entity_id
#+end_src

*** Architectural Requirements

Minimal.  The =OrderedSetAggregator= class already supports a =direct_argument= parameter.  Each new percentile is a one-line instantiation:

#+NAME: src:percentile-registration
#+begin_src python :exports code
p25 = OrderedSetAggregator(name='p25', aggregate='percentile_cont', direct_argument=0.25)
register_aggregation("p25", p25)
#+end_src

CV requires a new class similar to =Zscore= that combines =stddev= and =avg= in a single expression with a zero-division guard.

** P2: Inter-Event Interval Statistics

*** Description

Compute statistics over the time gaps between consecutive events for each entity: gap_mean, gap_stddev, gap_min, gap_max, gap_median, gap_cv.  These measure how regularly (or irregularly) events occur.

*** Why It Matters for Event Data

Inter-event intervals are the most natural feature family for event data and have no analogue in dense time series (where intervals are constant by definition).  A customer who orders every 7 days is fundamentally different from one who orders in bursts.  Gap statistics encode this behavioral pattern directly.  They are among the strongest predictors in churn models, fraud detection, and maintenance scheduling.

*** SQL Implementation

The approach uses a subquery that computes per-event gaps via =LAG=, then aggregates those gaps:

#+NAME: src:inter-event-gaps
#+begin_src sql :exports code
-- Inter-event interval statistics for each entity
select
    entity_id,
    avg(gap_days)                               as gap_mean,
    stddev(gap_days)                            as gap_stddev,
    min(gap_days)                               as gap_min,
    max(gap_days)                               as gap_max,
    percentile_cont(0.5)
      within group (order by gap_days)          as gap_median,
    case when avg(gap_days) = 0 then null
         else stddev(gap_days) / avg(gap_days)
    end                                         as gap_cv
from (
    select
        entity_id,
        extract(epoch from
            temporal_ix - lag(temporal_ix)
                over (partition by entity_id order by temporal_ix)
        ) / 86400.0 as gap_days
    from events
) gaps
where gap_days is not null
group by entity_id
#+end_src

*** Architectural Requirements

This requires a new =SubqueryAggregator= base class.  The current =Aggregator= generates a single aggregate expression; inter-event statistics need a preparatory subquery (the =LAG= computation) followed by aggregation over that subquery's output.  The planner would need to emit an additional CTE for the gap computation before the aggregation CTE.

** P3: Burstiness and Regularity Indices

*** Description

Three related features that characterize the temporal regularity of an entity's event stream:

- *Burstiness index* (Goh-Barabasi): $B = (\sigma_\tau - \mu_\tau) / (\sigma_\tau + \mu_\tau)$ where $\tau$ is the inter-event interval.  Ranges from -1 (perfectly periodic) through 0 (Poisson/random) to +1 (maximally bursty).
- *Event rate*: =COUNT(*) / time_span= --- events per unit time.
- *Time span*: =MAX(temporal_ix) - MIN(temporal_ix)= --- total active period.

*** Why It Matters for Event Data

Burstiness distinguishes entities with the same event count but different temporal patterns.  A facility with 12 inspections spread uniformly over 12 months behaves differently from one with 12 inspections in a single month (perhaps triggered by a compliance issue).  The burstiness index captures this in a single, scale-free number.  Event rate normalizes count by observation window.

*** SQL Implementation

#+NAME: src:burstiness
#+begin_src sql :exports code
-- Burstiness index and event rate
select
    entity_id,
    -- Burstiness (Goh-Barabasi)
    case
        when stddev(gap_days) + avg(gap_days) = 0 then null
        else (stddev(gap_days) - avg(gap_days))
             / (stddev(gap_days) + avg(gap_days))
    end as burstiness,
    -- Event rate (events per day)
    count(*) / nullif(
        extract(epoch from max(temporal_ix) - min(temporal_ix)) / 86400.0, 0
    ) as event_rate,
    -- Time span in days
    extract(epoch from max(temporal_ix) - min(temporal_ix)) / 86400.0
        as time_span_days
from (
    select
        entity_id,
        temporal_ix,
        extract(epoch from
            temporal_ix - lag(temporal_ix)
                over (partition by entity_id order by temporal_ix)
        ) / 86400.0 as gap_days
    from events
) gaps
group by entity_id
#+end_src

*** Architectural Requirements

Builds on the =SubqueryAggregator= infrastructure from P2.  The burstiness index is a derived expression over gap_mean and gap_stddev, so it could be implemented as a post-processing transformation on P2 outputs, or as a compound aggregator that emits the full expression.  Event rate and time span are simple aggregate expressions on the base entity and do not require the subquery pattern.

** P4: Shannon Entropy of Categorical Distributions

*** Description

Compute the Shannon entropy $H = -\sum p_i \ln(p_i)$ of categorical variable distributions within each entity's event history.  This measures the diversity or unpredictability of categorical event types.

*** Why It Matters for Event Data

Event data frequently contains categorical variables: complaint types, diagnosis codes, transaction categories, inspection outcomes.  Entropy measures whether an entity's events are concentrated in a few categories (low entropy --- predictable) or spread across many (high entropy --- diverse).  A patient who only visits for one diagnosis is different from one with diverse conditions.  This is one of the most informative features for event-categorical data and is absent from the current primitive set.

*** SQL Implementation

#+NAME: src:shannon-entropy
#+begin_src sql :exports code
-- Shannon entropy of event categories per entity
select
    entity_id,
    -sum(freq * ln(freq)) as entropy
from (
    select
        entity_id,
        category,
        count(*)::float / sum(count(*))
            over (partition by entity_id) as freq
    from events
    where category is not null
    group by entity_id, category
) category_freq
group by entity_id
#+end_src

*** Architectural Requirements

Requires the =SubqueryAggregator= pattern: a preparatory CTE computes per-category frequencies (using =COUNT(*) / SUM(COUNT(*)) OVER (PARTITION BY entity_id)=), then the outer query aggregates with =-SUM(p * LN(p))=.  The aggregator must specify which categorical variable to compute entropy over, which is a new parameter not present in the current =Aggregator= interface.  The input type should be =categorical=.

** P5: Concentration Indices (HHI and Gini)

*** Description

Two measures of concentration in categorical distributions:

- *Herfindahl-Hirschman Index (HHI)*: $\sum p_i^2$ --- ranges from $1/N$ (uniform) to 1 (all events in one category).
- *Gini coefficient*: measures inequality of the category distribution via cumulative shares.

*** Why It Matters for Event Data

Concentration indices complement entropy by quantifying market-style dominance in event categories.  An entity whose transactions are 90% in one category has high HHI, signaling behavioral concentration.  In supply chain data, high HHI for a store's supplier base indicates supply risk.  These features are standard in economics and transfer directly to entity-event feature engineering.

*** SQL Implementation

#+NAME: src:hhi-gini
#+begin_src sql :exports code
-- HHI: sum of squared category frequencies
select
    entity_id,
    sum(freq * freq) as hhi
from (
    select
        entity_id,
        count(*)::float / sum(count(*))
            over (partition by entity_id) as freq
    from events
    group by entity_id, category
) cat_freq
group by entity_id;

-- Gini coefficient via cumulative share
select
    entity_id,
    1.0 - 2.0 * sum(
        cum_share * freq
    ) / nullif(sum(freq), 0) as gini
from (
    select
        entity_id,
        freq,
        sum(freq) over (
            partition by entity_id
            order by freq asc
            rows between unbounded preceding and current row
        ) as cum_share
    from (
        select
            entity_id,
            count(*)::float / sum(count(*))
                over (partition by entity_id) as freq
        from events
        group by entity_id, category
    ) inner_freq
) ranked
group by entity_id
#+end_src

*** Architectural Requirements

HHI reuses the same frequency CTE as Shannon entropy (P4) --- the preparatory subquery is identical, only the final aggregation differs (=SUM(p*p)= instead of =-SUM(p*LN(p))=).  This argues for a shared =CategoryFrequencyCTE= building block.  Gini is more complex, requiring a nested window function for cumulative shares, but remains expressible as a single SQL query.  Both require the =SubqueryAggregator= pattern.

** P6: Cross-Entity Population Features

*** Description

Features that compare an entity's values to the population distribution across all entities:

- *cross_entity_zscore*: =(entity_value - population_mean) / population_stddev=
- *cross_entity_percentile*: =PERCENT_RANK() OVER (ORDER BY entity_value)= without =PARTITION BY=

*** Why It Matters for Event Data

Current transformations operate within an entity (partitioned by =entity_id=).  Population-level features answer a different question: "how does this entity compare to all others?"  A customer with 50 orders may be average in one business but exceptional in another.  Cross-entity z-scores and percentiles normalize entity-level aggregates against the population, providing essential context for ML models.

*** SQL Implementation

#+NAME: src:cross-entity
#+begin_src sql :exports code
-- Cross-entity z-score and percentile rank
select
    entity_id,
    total_amount,
    -- Population z-score (no PARTITION BY)
    (total_amount - avg(total_amount) over ())
        / nullif(stddev(total_amount) over (), 0) as pop_zscore_amount,
    -- Population percentile rank
    percent_rank() over (order by total_amount) as pop_percentile_amount
from (
    select entity_id, sum(amount) as total_amount
    from events
    group by entity_id
) entity_aggs
#+end_src

*** Architectural Requirements

Requires a new =PopulationWindowTransformer= class that emits window functions without =PARTITION BY= (or with =PARTITION BY= on a constant, which is equivalent).  This is architecturally distinct from the existing =WindowFunctionTransformer=, which always partitions by =parent.id=.  The new transformer would operate on already-aggregated entity-level features in the =_transform= CTE stage, applying population-level windowing.

** P7: Naive Change-Point Detection

*** Description

Simple proxies for detecting regime changes in an entity's event stream:

- *mean_shift_ratio*: ratio of the rolling mean over a recent window to the rolling mean over an older adjacent window.  Values far from 1.0 indicate a shift.
- *cusum*: cumulative sum of deviations from a target (often the overall mean), detecting sustained shifts.

*** Why It Matters for Event Data

Change-point detection is critical for event data: a facility that suddenly increases its inspection failure rate, a customer whose spending pattern shifts, a port whose dwell times jump.  While proper Bayesian change-point detection is iterative and outside SQL, these proxies capture the most common pattern --- mean shift --- with simple window arithmetic.

*** SQL Implementation

#+NAME: src:change-point
#+begin_src sql :exports code
-- Mean shift ratio: compare recent window to older window
select
    entity_id,
    temporal_ix,
    amount,
    avg(amount) over (
        partition by entity_id order by temporal_ix
        rows between 6 preceding and current row
    ) / nullif(
        avg(amount) over (
            partition by entity_id order by temporal_ix
            rows between 13 preceding and 7 preceding
        ), 0
    ) as mean_shift_ratio_7_14,
    -- CUSUM: cumulative deviation from overall entity mean
    sum(amount - avg(amount) over (partition by entity_id))
        over (
            partition by entity_id order by temporal_ix
            rows between unbounded preceding and current row
        ) as cusum_amount
from events
#+end_src

*** Architectural Requirements

Mean shift ratio requires two overlapping window specifications in a single expression, which the current =WindowFunctionTransformer= cannot produce (it generates one window per transformer).  A =CompoundWindowTransformer= or a custom class that emits the ratio of two window expressions would be needed.  CUSUM can be expressed as a nested window function (cumulative sum of deviations from a partitioned mean) and is closer to the existing =cum_sum= pattern but requires a two-pass computation.

** P8: Sequence Features for Categorical Events

*** Description

Features that capture sequential patterns in categorical event streams:

- *n_distinct_bigrams*: number of unique consecutive category pairs.
- *transition_entropy*: Shannon entropy of the bigram distribution.
- *most_common_bigram*: the most frequent category transition.

*** Why It Matters for Event Data

Event sequences carry information beyond marginal distributions.  A patient who transitions "ER -> ICU -> Discharge" has a different trajectory from "Routine -> Routine -> Routine", even if both have 3 events of similar categories in aggregate.  Sequence features capture first-order Markov structure in event streams and are among the most predictive features for behavioral modeling.

*** SQL Implementation

#+NAME: src:bigram-features
#+begin_src sql :exports code
-- Bigram features for categorical event sequences
with event_bigrams as (
    select
        entity_id,
        category || ' -> ' || lead(category)
            over (partition by entity_id order by temporal_ix)
            as bigram
    from events
    where category is not null
),
bigram_counts as (
    select
        entity_id,
        bigram,
        count(*) as cnt,
        count(*)::float / sum(count(*))
            over (partition by entity_id) as freq
    from event_bigrams
    where bigram is not null
    group by entity_id, bigram
)
select
    entity_id,
    count(distinct bigram) as n_distinct_bigrams,
    -sum(freq * ln(freq)) as transition_entropy,
    (array_agg(bigram order by cnt desc))[1] as most_common_bigram
from bigram_counts
group by entity_id
#+end_src

*** Architectural Requirements

This is the most architecturally complex extension.  It requires:

1. A preparatory CTE that computes bigrams via =LEAD= or =LAG= + concatenation.
2. A second CTE that counts bigram frequencies.
3. A final aggregation over the frequency distribution.

The planner currently emits at most two CTEs per entity (=_synth= and =_transform=).  Sequence features would need additional intermediate CTEs.  The =SubqueryAggregator= base class from P2 could be extended to support multi-step CTE chains, but the planner's CTE generation logic would need modification to accommodate the extra stages.

* Not Feasible --- And Why

The following feature families are fundamentally incompatible with the featurizer's constraints.  For each, we identify two independent reasons: one rooted in SQL limitations and one in the event data model mismatch.

** Autocorrelation and Long-Range Memory (ACF, PACF, Hurst Exponent)

*** What it is

Autocorrelation measures how correlated a signal is with lagged versions of itself.  The autocorrelation function (ACF) at lag $k$ is $\rho(k) = \text{Cov}(x_t, x_{t-k}) / \text{Var}(x_t)$.  The partial autocorrelation function (PACF) removes indirect correlations.  The Hurst exponent characterizes long-range dependence.

*** SQL limitation

ACF/PACF at arbitrary lags require iterative computation: for each lag $k = 1, \ldots, K$, compute the covariance with the $k$-lagged series.  While a single lag is a window function, producing the full correlogram requires $K$ passes or a recursive CTE with matrix operations.  PACF additionally requires solving a Yule-Walker system (matrix inversion), which has no SQL primitive.  The Hurst exponent involves log-log regression over multiple time scales, requiring nested aggregation that exceeds practical CTE complexity.

*** Event data mismatch

ACF assumes equispaced observations.  With irregular event timestamps, "lag 1" does not correspond to a fixed time offset.  The autocorrelation structure of an event process is not the same as that of a sampled signal.  Computing ACF on irregularly-spaced events either requires resampling to a regular grid (which destroys the event semantics and introduces interpolation artifacts) or using the Lomb-Scargle approach (which is itself iterative).

** Spectral Domain Features (FFT, Power Spectral Density, Spectral Entropy)

*** What it is

The Fourier transform decomposes a signal into frequency components.  Power spectral density (PSD) shows how power is distributed across frequencies.  Spectral entropy measures the uniformity of the PSD.

*** SQL limitation

The Fast Fourier Transform requires $O(N \log N)$ butterfly operations with complex arithmetic.  PostgreSQL has no built-in FFT.  While =pg_fft= extensions exist, they operate on arrays and are not expressible as standard SQL aggregates or window functions.  The entire spectral pipeline (windowing, FFT, magnitude computation, PSD estimation) is a multi-step array computation with no CTE-based equivalent.

*** Event data mismatch

Spectral analysis requires a fixed sampling frequency --- the Nyquist theorem, frequency bins, and spectral resolution all depend on equispaced samples.  Events have no natural sampling frequency.  Computing a spectrum of event timestamps (treating them as a point process) requires specialized techniques (Bartlett periodogram, multitaper spectral estimation for point processes) that are fundamentally different from standard FFT and have no SQL implementation.

** Wavelet and Multi-Resolution Analysis

*** What it is

Wavelet transforms decompose signals into time-frequency representations at multiple scales, using localized basis functions (wavelets).  Unlike Fourier analysis, wavelets provide time localization of frequency content.

*** SQL limitation

Wavelet transforms require iterative convolution with wavelet filter banks at each decomposition level.  The discrete wavelet transform (DWT) involves successive low-pass and high-pass filtering with downsampling --- operations that require array manipulation, filter coefficient storage, and recursive decomposition.  None of these have SQL equivalents.

*** Event data mismatch

Wavelets assume a regularly-sampled signal.  The "multi-resolution" in wavelet analysis refers to different time scales of a continuous signal, not to irregular event spacing.  Applying wavelets to event data would require resampling to a regular grid first, which introduces the same fundamental mismatch described for spectral methods.

** Nonlinear Complexity Measures (Approximate Entropy, Sample Entropy, Lempel-Ziv)

*** What it is

These measures quantify the regularity or complexity of a time series by examining patterns in subsequences.  Approximate entropy (ApEn) and sample entropy (SampEn) count template matches within tolerance $r$ across embedding dimension $m$.  Lempel-Ziv complexity counts the number of distinct subsequences.

*** SQL limitation

ApEn and SampEn require $O(N^2)$ pairwise distance comparisons between all $m$-dimensional embedded vectors.  This is a nested loop over the entire series for each point, which cannot be expressed as a single SQL aggregate or window function.  Even with a self-join approach, the quadratic complexity makes it impractical for typical event table sizes.  Lempel-Ziv requires sequential scanning with state (a dictionary of seen subsequences), which is inherently procedural.

*** Event data mismatch

Embedding dimension $m$ assumes consecutive samples at fixed intervals form a meaningful state vector.  For events, consecutive records may be separated by minutes or months.  The phase space reconstruction theorem (Takens' embedding) assumes continuous dynamics sampled at a fixed rate, which does not apply to discrete events.  The notion of "template matching within tolerance $r$" loses meaning when the underlying process generates categorical events at irregular times.

** Recurrence Quantification Analysis (RQA)

*** What it is

RQA analyzes recurrence plots --- binary matrices where entry $(i,j) = 1$ if the $m$-dimensional embedded vectors at times $i$ and $j$ are within distance $\epsilon$.  RQA metrics include recurrence rate, determinism, laminarity, and trapping time.

*** SQL limitation

Constructing the recurrence matrix requires the same $O(N^2)$ pairwise comparison as entropy measures.  Extracting diagonal and vertical line structures from the binary matrix requires pattern detection algorithms (connected component analysis, line counting) that are not expressible in SQL.

*** Event data mismatch

RQA inherits the phase space reconstruction assumption: it requires continuous dynamics sampled regularly.  Event sequences are categorical transitions, not continuous trajectories.  The geometric notion of "recurrence" (returning to a neighborhood in phase space) does not map onto event data, where the state space is discrete and the relevant notion of recurrence is categorical repetition, better captured by sequence features (P8).

** Graph and Topological Features (Visibility Graphs, Persistent Homology)

*** What it is

Visibility graphs convert time series into networks: two time points are connected if they can "see" each other (no intermediate value exceeds the line between them).  Persistent homology extracts topological features (connected components, loops) across multiple scales.

*** SQL limitation

Visibility graph construction requires $O(N^2)$ pairwise visibility checks.  Persistent homology requires building a filtration of simplicial complexes and computing Betti numbers, which involves linear algebra over Z/2Z fields --- entirely outside SQL's computational model.

*** Event data mismatch

Both techniques assume a continuous-valued signal with geometric structure.  Events lack the continuous amplitude needed for visibility calculations, and the topological features of a point cloud in "event space" are not meaningful in the way that topology of a continuous signal is.

** Learned Representations (TS2Vec, CPC, T-Loss)

*** What it is

Self-supervised learning methods that train neural networks to produce vector representations of time series, using contrastive learning objectives that do not require labels.

*** SQL limitation

Neural network inference requires matrix multiplication, nonlinear activation functions, and iterative optimization --- none of which can be expressed in a single SQL query.  While PostgreSQL extensions like =pgml= can run inference, they require pre-trained models and introduce a dependency outside the pure-SQL constraint.

*** Event data mismatch

Contrastive learning methods (TS2Vec, CPC) create positive pairs from overlapping windows of a dense series.  With sparse events, windows may contain zero or one event, making the contrastive objective degenerate.  These methods assume sufficient density to fill fixed-length input windows, which event data does not guarantee.

** Model-Based Residuals (ARIMA, GARCH, State Space)

*** What it is

Fitting parametric models (ARIMA for trend/seasonality, GARCH for volatility clustering, state space for latent dynamics) and extracting residuals, parameters, or forecasts as features.

*** SQL limitation

ARIMA fitting requires iterative maximum likelihood estimation (MLE) with gradient-based optimization.  GARCH similarly requires iterative MLE.  State space models (Kalman filter) involve recursive matrix operations.  None of these can be expressed as declarative SQL.  While the Holt-Winters approximation in the featurizer captures some trend/level structure, true model fitting is procedural.

*** Event data mismatch

ARIMA assumes a stationary (after differencing) regular time series.  The "I" in ARIMA (integration/differencing) is defined for equispaced observations.  GARCH models volatility clustering in equispaced returns.  Applying these to irregular events requires either resampling (destroying event semantics) or using point process models (Hawkes processes, Cox processes), which are different model families entirely.

* Borderline Cases

These features could work with preprocessing or architectural compromises, but sit at the boundary of feasibility.

** Markov Transition Probabilities

*** Description

For categorical event sequences, compute the full transition probability matrix: $P(Y_{t+1} = j \mid Y_t = i) = N_{ij} / N_{i\cdot}$.  Extract individual transition probabilities as features.

*** Why It Is Borderline

The SQL is straightforward (a self-join on lagged categories, grouped by origin-destination pair) and the event data model is a perfect fit.  The challenge is architectural: the number of features is $K^2$ where $K$ is the number of categories, which can be large.  The planner would need to enumerate all category pairs, or the user would need to specify which transitions matter.  The output is a matrix, not a scalar, which does not fit the current "one feature = one column" model.

#+NAME: src:markov-transition
#+begin_src sql :exports code
-- Transition probabilities for categorical sequences
select
    entity_id,
    prev_category,
    curr_category,
    count(*)::float / sum(count(*))
        over (partition by entity_id, prev_category) as transition_prob
from (
    select
        entity_id,
        category as curr_category,
        lag(category) over (
            partition by entity_id order by temporal_ix
        ) as prev_category
    from events
) lagged
where prev_category is not null
group by entity_id, prev_category, curr_category
#+end_src

*** What Would Be Needed

A =PivotAggregator= that can enumerate known categories (from a configuration parameter or a discovery query) and emit one column per transition.  Alternatively, a "top-N transitions" approach that limits output to the most common transitions in the training data.

** Stationarity Proxy

*** Description

Compare first-half and second-half statistics of an entity's event stream.  A large difference suggests non-stationarity (regime change).

*** What It Looks Like in SQL

#+NAME: src:stationarity-proxy
#+begin_src sql :exports code
-- Stationarity proxy: first-half vs. second-half mean ratio
with entity_midpoints as (
    select
        entity_id,
        min(temporal_ix) + (max(temporal_ix) - min(temporal_ix)) / 2 as midpoint
    from events
    group by entity_id
)
select
    e.entity_id,
    avg(case when e.temporal_ix <= m.midpoint then amount end) as first_half_mean,
    avg(case when e.temporal_ix > m.midpoint then amount end) as second_half_mean,
    avg(case when e.temporal_ix > m.midpoint then amount end)
        / nullif(avg(case when e.temporal_ix <= m.midpoint then amount end), 0)
        as mean_ratio
from events e
inner join entity_midpoints m on e.entity_id = m.entity_id
group by e.entity_id
#+end_src

*** Why It Is Borderline

The SQL is feasible but requires a join with a midpoint subquery, which adds a CTE.  The real limitation is that "first half" and "second half" are crude: they do not account for varying observation windows across entities, and the midpoint may fall in a gap with no events on one side.  The feature is a useful heuristic but not a principled stationarity test.

** Approximate Entropy Proxy via Rolling Variance Ratios

*** Description

Instead of true approximate entropy (which requires $O(N^2)$ comparisons), use the ratio of short-window variance to long-window variance as a regularity proxy.  Regular series have ratio near 1.0; complex series have higher ratios.

#+NAME: src:entropy-proxy
#+begin_src sql :exports code
-- Approximate entropy proxy: short/long rolling variance ratio
select
    entity_id,
    temporal_ix,
    amount,
    variance(amount) over (
        partition by entity_id order by temporal_ix
        rows between 6 preceding and current row
    ) / nullif(
        variance(amount) over (
            partition by entity_id order by temporal_ix
            rows between 29 preceding and current row
        ), 0
    ) as variance_ratio_7_30
from events
#+end_src

*** Why It Is Borderline

This is a standard window function composition, fully expressible in SQL and compatible with event data.  It is "borderline" because calling it an "entropy proxy" overpromises: it captures a specific aspect of regularity (short-term vs. long-term variance stability) but does not measure information-theoretic complexity.  It is, however, a useful feature in practice and could be implemented as a =CompoundWindowTransformer= alongside the change-point features in P7.

* Architectural Implications

Implementing the feasible extensions requires three new base classes and one planner modification.  This section summarizes the architectural roadmap.

** SubqueryAggregator Design

The =SubqueryAggregator= extends =Aggregator= with two methods instead of one:

1. =_build_preparatory_cte(feature, interval)= --- returns a CTE definition that computes intermediate values (gaps, frequencies).
2. =_build_aggregate_expression(feature, interval)= --- returns the final aggregate over the CTE output.

The planner's =_build_aggs_for()= method would check if an aggregator is a =SubqueryAggregator= and, if so, emit the preparatory CTE before the aggregation CTE.

** PopulationWindowTransformer Design

The =PopulationWindowTransformer= mirrors =WindowFunctionTransformer= but omits the =PARTITION BY= clause (or partitions by a constant).  It operates on the =_transform= CTE output, adding population-level comparisons.  The planner would need a new stage between =_transform= and the final =SELECT= to apply population transformations.

** Interval Filter Integration

The existing interval filtering (=FILTER (WHERE daterange(...) @> event_date)=) is specific to =Aggregator._build_aggregate_expression=.  =SubqueryAggregator= preparatory CTEs need to apply the interval filter at the subquery level (=WHERE temporal_ix >= as_of_date - interval '...'=) rather than the aggregate level.  This requires the interval to be passed through to the preparatory CTE builder.
