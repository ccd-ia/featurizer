* The Feature Engineering Bottleneck

Feature engineering consumes the majority of data scientist time in ML pipelines.

- Data scientists spend ~80% of their time on data preparation and feature engineering
- Manual SQL for features is repetitive, error-prone, and hard to maintain
- Each new prediction problem requires rewriting similar aggregation queries

* Relational Data is Everywhere

Most real-world ML operates on relational databases.

#+begin_example
Customers ──< Orders ──< OrderItems >── Products >── Suppliers
Patients  ──< Visits ──< Diagnoses
Students  ──< Enrollments ──< Grades
#+end_example

The features we need live in the /relationships/ between tables, not in any single table.

* The Manual Approach

#+begin_src sql
-- Just three features, already verbose and error-prone:
select c.customer_id,
       sum(o.amount) as total_spend,
       count(*) as order_count,
       avg(o.amount) as avg_order_value
from customers c
left join orders o on c.customer_id = o.customer_id
group by c.customer_id;
#+end_src

Now imagine: 5 aggregations \times 3 variables \times 4 time windows \times 20 transformations = 1,200 features...

* What Goes Wrong: Data Leakage

The most dangerous mistake in temporal ML: using future data to compute training features.

- A model trained on January 2025 must not see February orders when computing features
- Manual SQL makes this easy to get wrong --- a missing =WHERE= clause leaks the future
- Point-in-time correctness requires discipline that doesn't scale manually

* Point-in-Time Correctness

For each =as_of_date=, features must only use data available /at that point/:

#+begin_example
Timeline:  ──|────────|────────|────────|──→
          Jan 1    Feb 1    Mar 1    Apr 1

as_of = Feb 1:  ←─ visible ─→  [hidden]
                Jan data only    Feb/Mar/Apr invisible
#+end_example

Featurizer enforces this automatically via =CROSS JOIN LATERAL= against =as_of_dates=.

* What If We Could Automate This?

Given:

- An entity-relationship graph (YAML config)
- A set of feature primitives (aggregations + transformations)
- Temporal semantics (as-of dates, intervals)

Generate: hundreds of point-in-time correct features as pure SQL.

That is *Featurizer*.

* Why Not Featuretools?

Featuretools (Python DFS library) works well for in-memory data, but:

- *Data movement*: Loads tables into Python memory --- fails on large datasets
- *No temporal guarantees*: Point-in-time correctness is opt-in, easy to miss
- *No SQL output*: Cannot inspect, audit, or deploy the generated logic
- *Single-node*: Cannot leverage PostgreSQL's query optimizer and parallelism

Featurizer generates pure SQL --- data never leaves the database.

* Deep Feature Synthesis: The Core Idea

DFS (Kanter & Veeramachaneni, 2015) automates feature engineering by:

1. Traversing the entity graph
2. At each edge: applying aggregations (backward) or lookups (forward)
3. At each node: applying transformations
4. Composing these across multiple depths

#+begin_example
Entity Graph:     Customers ←── Orders ←── Items
DFS Traversal:    ← aggregate   ← aggregate
                  transform     transform
#+end_example

* Forward vs Backward Relationships

Two traversal directions yield different feature types:

- *Backward* (parent \leftarrow child): Aggregate many child rows into one parent value
  - =SUM(orders.amount)=, =COUNT(orders.status)=
- *Forward* (child \rightarrow parent): Look up parent attributes
  - =customer.country=, =customer.age=

Backward relationships are where the richness lives.

* DFS: The Algorithm

The original DFS algorithm from Kanter & Veeramachaneni (DSAA 2015):

1. For each entity in the graph:
2. Collect base features (declared variables)
3. For each backward relationship: aggregate child features
4. For each forward relationship: look up parent features
5. Apply transformations to all features
6. If =depth < max_depth=: recurse into related entities

Featurizer implements this in pure SQL, producing CTEs at each step.

* Three-Stage Pipeline

#+begin_example
┌──────────┐     ┌──────────────┐     ┌──────────┐
│ Planner  │ ──→ │ SQL Renderer │ ──→ │ Executor │
│          │     │              │     │          │
│ DFS      │     │ CTEs +       │     │ Run SQL  │
│ traversal│     │ lateral join │     │ → DataFrame│
└──────────┘     └──────────────┘     └──────────┘
     ↑                                      │
  YAML config                          pandas DataFrame
#+end_example

All three stages are independently importable for custom workflows.

* The Planner

The planner performs DFS traversal of the entity graph:

1. Start at the target entity
2. Traverse backward relationships \rightarrow generate aggregation CTEs
3. Traverse forward relationships \rightarrow generate lookup CTEs
4. At each entity, apply transformations
5. Produce two CTEs per entity: =<alias>_synth= and =<alias>_transform=

The planner is /pure logic/ --- it produces CTE definitions, not SQL strings.

* CTE Generation Pattern

For each entity, two CTEs are generated:

#+begin_src sql
-- Stage 1: Synthesize aggregated + direct features
<entity>_synth AS (
  SELECT base.*, aggs.feature1, aggs.feature2
  FROM <entity> base
  LEFT JOIN <child>_aggs_for_<entity> aggs ON ...
),

-- Stage 2: Apply transformations
<entity>_transform AS (
  SELECT *, LAG(feature1) OVER (...) AS lag_1_feature1,
            AVG(feature1) OVER (...) AS rolling_mean_7_feature1
  FROM <entity>_synth
)
#+end_src

* The SQL Renderer

The renderer wraps CTEs in the temporal execution context:

#+begin_src sql
SELECT aod.as_of_date, t.*
FROM as_of_dates AS aod
CROSS JOIN LATERAL (
  WITH
    orders_synth AS (...),
    orders_transform AS (...),
    orders_aggs_for_customers AS (...),
    customers_synth AS (...),
    customers_transform AS (...)
  SELECT * FROM customers_transform
) AS t
ORDER BY aod.as_of_date
#+end_src

The =CROSS JOIN LATERAL= ensures point-in-time correctness.

* Temporal Joins (As-Of Semantics)

Relationships can declare temporal blocks:

#+begin_src yaml
temporal:
  mode: as_of
  grace: P21D
  child_timestamp: custom_field
#+end_src

Generated SQL uses =LEFT JOIN LATERAL= to pull the most recent parent record:

#+begin_src sql
LEFT JOIN LATERAL (
  SELECT * FROM parent
  WHERE parent.id = child.parent_id
    AND parent.timestamp <= child.timestamp
    AND parent.timestamp >= child.timestamp
        - INTERVAL 'P21D'
  ORDER BY parent.timestamp DESC
  LIMIT 1
) AS parent_asof ON TRUE
#+end_src

* The SubqueryAggregator Architecture

Some aggregations require correlated subqueries:

#+begin_src sql
-- Gap statistics need LAG() which can't nest
-- inside GROUP BY. SubqueryAggregator generates:
(SELECT AVG(gap) FROM (
  SELECT EXTRACT(EPOCH FROM
    ts - LAG(ts) OVER (ORDER BY ts)
  ) AS gap
  FROM child_table
  WHERE child_key = parent.parent_key
) sub WHERE gap IS NOT NULL)
#+end_src

Used by: =gap_mean=, =gap_stddev=, =entropy=, =hhi=, =gini=, =ngram_2_freq=, =sequence_entropy=, =longest_streak=

* Gap Statistics and Burstiness

Inter-event gap analysis for temporal patterns:

#+begin_example
Events:   ●──────●──●●──────────●───●
Gaps:        6d   1d 0d    8d     2d

gap_mean = 3.4 days
gap_cv = 0.95 (high variability)
burstiness = (σ-μ)/(σ+μ) = 0.42 (moderately bursty)
#+end_example

- =burstiness = -1=: perfectly regular (clock-like)
- =burstiness = 0=: random (Poisson)
- =burstiness = 1=: highly bursty (clustered)

* The SubqueryAggregator Pattern

#+begin_src python
class SubqueryAggregator(Aggregator):
    """Base class for aggregations requiring
    correlated subqueries within GROUP BY."""

    def __call__(self, relationship, feature, interval):
        # Access child table alias and join key
        child_alias = relationship.child.alias
        child_key = relationship.child_key
        # Generate correlated subquery SQL
        return self._build_subquery(
            feature, child_alias, child_key
        )
#+end_src

The planner passes =relationship= to aggregator calls, enabling SubqueryAggregator to correlate.

* Population Windows

Cross-entity statistics normalize across /all/ entities:

#+begin_src sql
-- cross_entity_zscore: How unusual is this entity?
(value - AVG(value) OVER ())
  / NULLIF(STDDEV(value) OVER (), 0)

-- cross_entity_percentile: Where does this entity rank?
PERCENT_RANK() OVER (ORDER BY value)
#+end_src

Window =OVER ()= spans the entire result set, comparing each entity to the population.

* Change-Point Detection

Detect regime changes in feature behavior:

#+begin_src sql
-- mean_shift_ratio: Compare recent window to overall
AVG(val) OVER (... ROWS 6 PRECEDING)
  / NULLIF(AVG(val) OVER (), 0)

-- CUSUM: Cumulative sum of deviations
SUM(val - target_mean)
  OVER (PARTITION BY id ORDER BY date)
#+end_src

- =mean_shift_ratio \approx 1.0=: stable
- =mean_shift_ratio \gg 1.0=: recent increase
- CUSUM drift: sustained shift that rolling averages miss

* Extensibility: Custom Primitives in 10 Lines

#+begin_src python
from featurizer.primitives.aggregations import Aggregator
from featurizer.primitives.utils import register_aggregation

class WeightedSum(Aggregator):
    def __init__(self):
        super().__init__(name="weighted_sum")

    def _build_aggregate_expression(self, feature, interval):
        base = super()._build_aggregate_expression(
            feature, interval
        )
        return base.replace(
            feature.name, f"{feature.name} * weight"
        )

register_aggregation("weighted_sum", WeightedSum())
#+end_src

Register once, use everywhere via config.

* Performance Considerations

Featurizer generates SQL --- performance depends on PostgreSQL:

- *Indexes*: Ensure =temporal_ix= columns have B-tree indexes
- *Partitioning*: Large fact tables benefit from range partitioning on time
- *Parallelism*: PostgreSQL parallel query handles aggregation CTEs well
- *Materialization*: Use =CREATE TABLE AS= for large feature matrices

Rule of thumb: if the query runs in PostgreSQL, Featurizer generates it correctly.
