Purpose

Make featurizer's full-aggregator (all-agg) feature matrix materialize in practical time on real entity-event data. Today it is dominated by child-stream fan-out: the planner emits one companion CTE per (family, column, interval), each independently scanning the child <child>_transform stream, all merge-joined into a ~100-way join. This plan makes the set-based pre-aggregation pre-passes composable and merges those that share a window signature, so the child stream is scanned a handful of times and the join width collapses — without changing a single output value or column name (ADR-0007 / ADR-0010 contract), proven by the frozen golden-value gate.

Problem

Measured on the live triage DBs (see specs/feature-materialization-performance.html): dirtyduck all-agg = 356.8s, donorschoose = 281.1s. Diagnostics proved this is not a slow primitive (every family in isolation runs in 1–8s), not a planner-collapse tuning problem (conservative tuning bought only 1.4×; aggressive tuning crashed the backend), not fixable by column-sharding or parallelism (one column-group holds 343 of 350s), and not the as_of_dates cardinality estimate (ANALYZE did nothing). The EXPLAIN + string counts show the cause directly, and it is general, not a dirtyduck quirk — the fan-out grows with config richness on every DB:

DBfeaturestotal CTEscompanion CTEschild-stream refsall-agg time
chicago3111914812632410.1s
dirtyduck272105303813356.8s
donorschoose10632795822091281.1s

There are zero residual correlated subqueries (ADR-0010's migration is complete). The cost is pure fan-out: cost ≈ (# companion CTEs) × (child-stream size) × (join-width penalty), and all three factors grow with entities, events, history depth, categorical columns, intervals, and the aggregator set. chicago311 is fast only because it sits at the low end of every axis.

Solution

Give every set-based aggregator a composable pre-pass contract: instead of an opaque full prepass_sql subquery, a family declares (1) a window signature — the child stream, the PARTITION BY group key, the ORDER BY, and the interval — and (2) the derived per-row columns it needs (its window/scalar expressions, uniquely aliased). The planner then groups every family sharing a signature and emits one pre-pass per (signature, interval) that computes the union of all derived columns in a single windowed scan, wrapped by one GROUP BY producing every family's reduction. Reductions are unchanged, so the output columns are byte-identical; the fixed golden values still hold. Fan-out drops from ~100 scans to roughly one per window signature per interval.

Notes

Why not P1 / P2 (already measured, do not re-litigate)

The parent spec specs/feature-materialization-performance.html records the full evidence: P1 (conservative work_mem/collapse tuning) buys only ~1.4× and is a supporting default at best (GEQO must stay ON — geqo=off crashed the backend). P2 (bounded-CTE sharding + parallelism) was refuted by measurement: sharding keeps serial total constant (~350s) and one column-group holds 343 of 350s, so parallelism can't help either. P3 is the only lever that attacks the actual cost (the fan-out). This plan is P3.

Design alternatives & open decisions

  • Option A vs B sequencing. A (shared windowed base, Phase 2) is a contained scan-cost win that ships independently and de-risks the plumbing; B (signature merge, Phase 3) is the structural collapse. If Phase 2's delta proves marginal on its own, it still stands as the pre-filter the merged pre-passes read from — the two compose. Decision: keep both, A first.
  • Contract shape. Preferred: families expose (alias, sql_expr) derived columns + a signature (composable), so the planner can union them. Rejected: keeping opaque prepass_sql and string-diffing to detect mergeable pre-passes — brittle and unauditable.
  • Incrementality. Migrate family-by-family behind the golden gate (ADR-0010's proven strategy). Three CTE paths (correlated / legacy-opaque / composable-merged) coexist until every family migrates — never a big-bang cutover.
  • Merge boundary. Only families with an equal window signature merge. Different PARTITION BY/ORDER BY (e.g. a numeric-stream family vs a categorical family over a different column) stay separate — correctness first, coverage second.
  • Interval nesting (future, out of scope). Deriving narrower intervals from the widest window in one pass is a further win but a separate effort; not in this plan.

Risks & mitigations

  • Silent value drift → the golden-value gate after every family (162 frozen cases, exact ints / FP-tol floats). A mismatch stops the line; fix the rewrite, never the fixture.
  • Name/contract drift (ADR-0007 / triage-pg) → output column names come from unchanged _build_name/_build_label; the collision snapshot guards it. Merged-CTE names are internal only (and must be valid bare identifiers — mind the ~-cap fix already in planner.py).
  • Sharding interaction → merged CTEs register with ShardableCTE/MaterializationKey like today; per-column synth pruning must still drop a shard's unused reductions. Covered by test_preagg_shape.py + the sharding tests. (Also fold in the manifest group_000-fallback assertion noted in the parent spec.)
  • Planner complexity → new merge surface in planner.py; keep the legacy path bit-for-bit so un-migrated families are provably untouched.

Acceptance target

Dirtyduck all-agg materialization well under 120s on the real DB (ADR-0010's intent, which was only ever met on synthetic 10k data), with the golden values unchanged and output column names byte-identical. Companion-CTE count for dirtyduck reduced from ~100 toward ~(distinct signatures × intervals). Donorschoose (281s) and chicago311 (already 10s) improve or hold.

2026-07-09 — Phase 1 built & green; Phase 3 premise revised mid-build (family heterogeneity)

Phase 1 — DONE. benchmarks/fanout_report.py + just bench-fanout + the DB-free budget guard tests/test_fanout_budget.py (synthetic 3-categorical / 1-numeric all-agg config = 132 companion CTEs, pinned). Golden gate confirmed green (163 passed) and DB-free suite green (576). The harness that de-risks the rest is in place.

Phase 2 — DEFERRED (evidence). The per-group EXPLAIN shows the cost is the ~100-way merge-join chain, not the per-scan filter. Option A (shared per-interval windowed base) lowers per-scan cost but leaves join width unchanged, so it does not attack the measured bottleneck. Not built.

Phase 3 — PREMISE REVISED, paused for a decision. Reading the real family _build_preagg bodies revealed the pre-passes are genuinely heterogeneous, so "one merged pre-pass per (signature, interval)" cannot collapse ~100 CTEs to a handful: theil filters to col>0 (different row-set); mad is a join to a per-group median (different shape); catfreq is GROUP BY category (different granularity); gini needs order by val while meandev needs no order. Only families with an identical source + WHERE + granularity merge (e.g. gini+meandev), yielding a partial fan-out reduction, not the dramatic collapse the plan assumed. Since the join width ≈ number of companion CTEs and that number is only partly reducible, pre-pass merging alone will not remove the join-width bottleneck.

DECISIVE CORRECTION (2026-07-09, later) — the fan-out was a red herring. An EXPLAIN (ANALYZE) of the 343s group (which I should have run first) shows the entire cost is 9 correlated SubPlans over inspections_transform at loops=18909 — i.e. correlated subqueries re-executed once per target row, ~34s each. These are the two-window drift families (kl_drift, wasserstein_drift) that ADR-0010 explicitly deferred as a non-goal — but they fire on dirtyduck's 3 categorical columns × 3 intervals (2×3×3 = 18 correlated subqueries) and dominate. Measured proof: dirtyduck all-agg MINUS {kl_drift, wasserstein_drift} = 25.2s (vs 356.8s). The set-based fan-out is cheap (66-CTE group = 5s). So neither P3 (fan-out) nor the union-pivot idea is the fix — migrating the 2 correlated drift families to set-based (the ADR-0010 pattern) is, and it is far smaller than this whole plan. This plan's premise is retired; see the new direction below. RESOLVED (2026-07-09): the 2 drift families were migrated to set-based (ADR-0012, option b) — dirtyduck all-agg 356.8s → 27.6s (~13×), all 272 features kept, values proven identical by the golden gate (233 passed). The fan-out harness from Phase 1 stays as a useful metric; the rest of this plan is not needed.

Retired direction (kept for the record): (a) Partial merge — implement the composable contract and merge the compatible clusters (gini/meandev-style); modest, well-understood, golden-gated win. Or (b) Eliminate the wide join — restructure the ~100 LEFT JOIN companion CTEs into a long-format UNION ALL of (key, feature, value) reduced by one GROUP BY pivot, sidestepping the 100-way join entirely. Potentially the real win, but a larger rearchitecture than this plan scoped. Recommend prototyping (b) on dirtyduck before committing either.
