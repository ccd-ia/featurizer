Purpose

Stress-testing this featurizer tree against triage-pg's three live databases (dirtyduck/food, chicago311, donorschoose) proved the just-landed transformer-label fix scales cleanly — zero name collisions and every identifier ≤ 63 bytes up to 52,225 features — but it also surfaced four pre-existing bugs that make the wider primitive sets unusable on real schemas. This plan fixes those four bugs, re-validates every variant against the live databases, and delivers a polished per-database HTML artifact that documents the config, the database structure, and a representative sample of the generated features (name, short-name, and a plain-language explanation).

Problem

triage-pg deliberately runs featurizer with a narrow primitive set (count/sum/mean/max/stddev + identity) — the config comments say the full set "blows past PostgreSQL's 1664-column limit" and produces "deeply-nested colliding names." Widening the set to exercise the engine surfaced four independent defects. All predate the label fix; they only appear when a non-curated primitive touches a column type it does not support.

The four defects, annotated & risk-rated

Medium B1 · EVENT_RATE — EXTRACT(EPOCH FROM integer)

Surfaced on: dirtyduck/food (all-agg, wide) · would also hit donorschoose · featurizer/primitives/aggregations.py:496-504

Emitted SQL
count(*) / NULLIF(EXTRACT(EPOCH FROM max(date) - min(date)), 0)
Root cause
On a date-typed column, max(date) - min(date) is an integer (day count), and EXTRACT(EPOCH FROM <integer>) is undefined — hence function pg_catalog.extract(unknown, integer) does not exist. The primitive assumes a timestamp column (where the difference is an interval).
Fix
Make it type-agnostic and report events per day: count(*) / NULLIF((EXTRACT(EPOCH FROM max(date)) - EXTRACT(EPOCH FROM min(date))) / 86400.0, 0). EXTRACT(EPOCH FROM …) returns numeric seconds for both date and timestamp; dividing by 86400 yields days (the interpretable, EMA-consistent unit — see Q2).
Why medium
Contained to one primitive; it was fully broken on date columns (no regression risk), but the SQL must stay correct for the timestamp case and interact correctly with the interval filter (where …) clause.

Medium B2 · BURSTINESS + gap_* family — STDDEV(interval)

Surfaced on: chicago311 (events.date :: timestamp) · aggregations.py:704-751, 771-782 — shared by gap_mean/gap_stddev/gap_min/gap_max/gap_cv/burstiness

Emitted SQL
SELECT (STDDEV(gap) - AVG(gap)) / … FROM (SELECT sub.date - LAG(sub.date) OVER (…) as gap …)
Root cause
timestamp - timestamp = interval. AVG/MIN/MAX accept interval, but STDDEV(interval) is undefined → the error. (On date columns the gap is integer days and it happens to work — so the bug is type-dependent and silent until a timestamp dataset hits it.)
Fix
Difference epoch days, not raw temporal values: (EXTRACT(EPOCH FROM sub.date) - LAG(EXTRACT(EPOCH FROM sub.date)) OVER (ORDER BY sub.date)) / 86400.0 → numeric gap in days. All six aggregators then work on both date and timestamp.
Why medium
Touches six primitives sharing one helper. burstiness/gap_cv are dimensionless ratios (the /86400 cancels — semantics unchanged, just now valid); gap_mean/min/max emit days, which matches the original integer-day output on date columns (no change there) and merely normalizes timestamp columns from an invalid interval to fractional days. Not in any triage config (default set is 10 curated aggregations; the gap family is opt-in) → no cross-repo contract impact.

High B3 · GEOMETRIC_MEAN — malformed SQL + wrong log base

Surfaced on: donorschoose (all-agg) · aggregations.py:279-289

Emitted SQL
( case when x > 0 then exp(avg(log(x)) else (-1.0)^count(*)*exp(avg(log(abs(x))) end )
Root cause
Two defects: (1) unbalanced parentheses — exp(avg(log(x)) closes log and avg but leaves exp( open, so PostgreSQL hits else mid-expression → syntax error at or near "else"; (2) it uses log() which is base-10 in PostgreSQL, whereas the geometric mean needs the natural log: exp(avg(ln(x))). The negative-value branch (-1)^count·exp(avg(ln|x|)) is non-standard and undefined for a true geometric mean.
Fix
Positive-domain definition (mirrors the existing CumProd transformer): case when min(x) > 0 then exp(avg(ln(x))) else null end.
Why high
Rated high for correctness: the current expression never produced valid SQL (so zero regression risk), but the replacement must be mathematically right and NULL-safe. A wrong fix would silently emit plausible-but-incorrect feature values — worse than a crash.

Low B4 · in_array — planner arity crash

Surfaced on: every max variant · transformations.py (IsInArray)

Symptom
IsInArray.__call__() missing 1 required positional argument: 'an_array'
Root cause
in_array is registered in DEFAULT_TRANSFORMERS, but the planner applies transformers as transformer(entity, feature). IsInArray.__call__ requires a third an_array argument the config format can't supply, so it crashes whenever the default set is used wholesale.
Fix
De-register in_array from DEFAULT_TRANSFORMERS (keep the class for explicit, argument-supplying callers). Optionally, harden the planner to skip transformers whose __call__ arity ≠ 2.
Why low
Isolated, niche primitive; the fix is a one-line de-registration with an obvious blast radius. It blocks the full-transform set today, so value is high even though risk is low.
Non-goal (documented ceiling, not a bug): the full max set (67 aggregations × 82 transformations, nested) is a combinatorial explosion — donorschoose blew past a 60 s build timeout and "wide" already reaches 52,225 features at depth 3. This plan does not try to make max practical; it is noted for a future depth/primitive-budget guard (see Notes).

Solution

Three surgical primitive fixes plus one de-registration, then re-validation against the three live databases — materializing the wide/over-1664-column configs through featurizer's sharded path and timing it (not skipping them) — then the deliverable artifacts. The unifying idea for B1/B2 is "extract epoch, express in days, never subtract raw temporal values" — a single type-agnostic rule (EXTRACT(EPOCH FROM col)/86400.0) that makes every temporal primitive correct on both date and timestamp columns and interpretable in days.

Each fix ships with an integration test that executes the primitive against real PostgreSQL over both a date and a timestamp temporal column, plus a DB-free scaled-collision regression test that locks in the transformer-label fix at 1000s of features. Then Phase 3 renders one self-contained HTML artifact per database.

Questionables

Q1 — geometric_mean semantics for non-positive inputs — RESOLVED: return NULL

Return NULL when min(col) ≤ 0 (geometric mean is undefined on non-positive values). Mirrors the existing CumProd transformer and is NULL-safe for the imputation layer. Signed pseudo-geometric-mean rejected as non-standard/misleading. (Confirmed by user, 2026-07-06.)

Q2 — gap-statistic units — RESOLVED: days, not seconds

Emit gaps and spans in days (EXTRACT(EPOCH FROM col)/86400.0), not seconds. Decisive reason: the original working output on date columns was already integer days, so days preserves existing date-column semantics exactly — only timestamp columns change (invalid interval → fractional days). Days is also interpretable and matches featurizer's EMA transformer (…/86400.0). For the dimensionless ratios (burstiness, gap_cv) the divisor cancels. Note: "triage doesn't use the gap family" means the default aggregation set is 10 curated primitives (count/max/mean/median/min/nunique/recency/stddev/sum/tenure); gap_*, event_rate, burstiness, geometric_mean are opt-in extras in the 67-set that no triage config selects. (Confirmed by user, 2026-07-06.)

Q3 — release packaging — RESOLVED: fold into v0.5.1

Fold these four fixes together with the unreleased transformer-label work into a single v0.5.1 patch via the existing release.yml pipeline. (Confirmed by user, 2026-07-06.)

Cross-repo safety

triage-pg pins featurizer at v0.4.2. Its configs use either an explicit narrow set (count/sum/mean/max/stddev + identity, for chicago311 & donorschoose) or the curated 10-primitive default (count/max/mean/median/min/nunique/recency/stddev/sum/tenure, for dirtyduck). Neither includes event_rate, the gap_* family, burstiness, or geometric_mean — those are opt-in extras in the 67-primitive full registry. So B1–B4 fixes are backward-safe for the downstream one-hot contract (ADR-0007) and change no name or value that triage consumes.

  • Transformer-family label truncation is already fixed (this session); these primitive fixes are the aggregation-side follow-up.
