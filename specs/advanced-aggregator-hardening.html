Purpose

v0.5.1 fixed the specific advanced-aggregator bugs the triage-dataset stress test happened to reach (temporal epoch handling, geometric_mean, skewness/kurtosis). But it also revealed the root problem: featurizer's ~57 advanced aggregators (everything beyond the curated default-active set) were only string-shape tested — no test ever executed their generated SQL on real data — so they accumulated bugs that only appear at runtime: division-by-zero, degenerate single-row groups, and a planner CTE defect. This plan closes that gap systematically: build execution coverage for the whole registry first (measure, don't guess), then fix everything it surfaces, and make an explicit decision about the correlated-subquery performance ceiling.

Problem

Five documented defects (from the v0.5.1 session), all rooted in "never executed on real data":

High P1 · No execution coverage for the advanced set (the root cause)

Evidence
tests/integration/test_default_primitives_execution.py deliberately covers only the default-active set; test_default_active_set_is_fully_covered records the advanced primitives as knowingly-excluded. So no test executes them — string-shape tests miss dialect/runtime bugs (exactly how bug #6, the ge/last/rolling bugs, and now this cluster all slipped through).
Fix
A full-registry execution harness that runs every registered aggregator against a fixture engineered to trigger edge cases (below). This is the foundation — it produces the authoritative broken-list instead of us guessing.

High P2 · Unguarded divisions → division-by-zero on real data

Evidence
donorschoose all-agg materialization raised division by zero at runtime (the SQL is valid — it only fails on data). Static scan flags ≥ 1 unguarded division in harmonic_mean (1.0/x with x=0), entropy, theil, cosinor_amplitude_weekly, markov_conditional_entropy, sequence_entropy, max_transition_prob (+ the excluded z_score/min_max_scale/mean_deviation).
Fix
Wrap every denominator in NULLIF(denom, 0); for harmonic_mean guard the reciprocal (1.0/NULLIF(x,0)). Zero-denominator → NULL, never a crash.

Medium P3 · Degenerate inputs (single-row / constant groups)

Evidence
A group with one event has undefined stddev/gap/skew; a constant column has var_pop = 0 (already NULLIF-guarded in the new moments, but many others aren't). These must return NULL (undefined), never error or emit garbage.
Fix
Decide and apply a uniform degenerate-input contract (NULL) across the stat aggregators; the Phase-1 harness includes 1-row and constant-column fixtures to prove it.

Medium P4 · Planner empty-CTE bug

Evidence
A single-type aggregation set (e.g. [entropy]) over a mixed-type entity graph emits select <key>,  from … — a dangling comma — for the entity where the aggregator produced no features (the 12 "syntax error at or near from" enumeration hits). It does NOT occur in the default/all-agg set (other aggregations populate the CTE), but it is a real robustness bug.
Fix
In the planner, skip emitting an <child>_aggs_for_<target> CTE (or its projection) when the aggregation yields zero features for that child.

Low P5 · Mis-classified primitives (mean_deviation, z_score, min_max_scale)

Evidence
v0.5.1 removed all three from the default set. z_score/min_max_scale are per-row normalizations (bare column, invalid as a GROUP BY reduction); mean_deviation (MAD) is a genuine reduction but nests aggregates.
Fix
Rewrite mean_deviation as a two-pass SubqueryAggregator and re-add to defaults; reclassify z_score/min_max_scale as transformers OR formally deprecate them (redundant with cross_entity_zscore/cross_entity_percentile).

High P6 · Correlated-subquery scaling (architectural)

Evidence
26 aggregators are SubqueryAggregators — a correlated subquery per target row via the lateral join. all-agg materialization over 22k–30k target rows exceeds 300 s (O(target-rows × subqueries × child-scan)). The narrow set materializes in seconds; the advanced set does not scale to full-cohort materialization.
Fix
A spike + decision, not necessarily a full rewrite: characterize the cost, evaluate pre-aggregating the child stream once per group vs. LATERAL rewrites vs. accepting the ceiling, and record the decision as an ADR. High risk = high effort/uncertainty; may spawn its own implementation plan.

Solution

Measure first. The through-line of the v0.5.1 session is that string-shape tests hid every one of these bugs. So Phase 1 builds the execution harness that makes "every registered aggregator executes on real data (over numeric / categorical / date / timestamp columns, and degenerate groups) without error" a tested invariant. That harness produces the authoritative broken-list; Phases 2–5 fix what it finds, in correctness-first order, ending with an explicit decision on the performance ceiling.

Questionables

Q1 — Degenerate-input semantics — RESOLVED: NULL

Return NULL for genuinely-undefined results (stddev/skew/gap over a single row, a ratio with a zero denominator). NULL is honest, composes with the imputation layer, and matches the moment fix already shipped. (Confirmed by user, 2026-07-06.)

Q2 — z_score / min_max_scale — RESOLVED: drop

Drop the aggregator classes (formal deprecation): they are per-row normalizations already covered by the cross_entity_zscore / cross_entity_percentile transformers. A partition-relative z-score, if ever wanted, becomes a new clearly-named transformer — not a resurrection of the mis-classified aggregators. (Confirmed by user, 2026-07-06.)

Q3 — Correlated-subquery scaling — RESOLVED: spike-and-defer

Spike-and-defer. Phase 5 characterizes the cost and records an ADR; the actual O(rows×subqueries) remediation gets its own dedicated plan later. Correctness (Phases 1–4) is the commitment here. (Confirmed by user, 2026-07-06 — "we will create a plan for this later".)

Why this exists

featurizer's execution-coverage test (test_default_primitives_execution.py, issue #6) was scoped — by an explicit decision — to the default-active set, recording the advanced primitives as knowingly excluded "so there are no silent gaps." The gap was documented but never closed; the advanced tier kept accumulating never-executed bugs. This plan closes it: after Phase 1, "every registered aggregator executes on real data" is a tested invariant, and the string-shape-only blind spot is gone for good.

Scope discipline

Phases 1–4 are correctness and are the commitment. Phase 5 is a spike whose deliverable is a decision (ADR), not a rewrite. This keeps the plan bounded and avoids the exact scope-explosion that made the v0.5.1 session long — the correctness surface is finite and enumerable; the performance surface is open-ended and gets a decision gate, not an open build.
