Purpose

Fix the two relationship-topology bugs discovered in the 2026-07-03 session — (1) aggregation relationships whose parent_key ≠ child_key render invalid SQL, and (2) parallel relationships between the same entity pair silently collapse into one feature set — and finish the feature-manifest story: persist the manifest as a table next to the feature-group tables, export the lineage the planner already tracks in memory (Feature.parents, stack_depth), and generate human-readable descriptions from existing primitive metadata. Ships as v0.5.0 (new config surface + new persisted artifact).

Problem

Both bugs were verified empirically against v0.4.2 (rendered SQL inspected; probe configs in the 2026-07-03 session):

Bug 1 — key-name mismatch renders invalid SQL. _build_aggregations_cte (planner.py:1216–1226) projects and groups by {source}_transform.{parent_key} — a column that only exists on the parent — while the join it emits (planner.py:1199–1201) references cte.{child_key}, a column the CTE never outputs. With parent: customers.customer_id / child: orders.buyer_id, the CTE says select orders_transform.customer_id … group by customer_id (no such column) and the synth joins on orders_aggs_for_customers.buyer_id (not projected). Two sites, mutually inconsistent, both wrong. _build_direct_cte (planner.py:1283–1300) has the same disease in the forward direction: it projects source.id.name but joins cte.{child_key}. Every shipped example and fixture happens to use parent_key == child_key == id, which is the only reason CI is green.

Bug 2 — parallel relationships silently collapse. Feature names are built from feature.entity.alias only (aggregations.py:96–101 _build_name) and the aggregation CTE is named by entity pair ({source}_aggs_for_{target}, planner.py:1198). Declaring customers→orders twice (buyer role, seller role) produced 5 features instead of 10, one CTE, and one join — the seller-side features vanish with no warning. Identity-collapse sites: Feature.__eq__ is name-based, aggregation_set = set(...) (planner.py:1042), _synth_column_source keyed by feature name (planner.py:1257–1259), and the traversal guard if child in self._path: continue (planner.py:348).

Gap 3 — manifest is in-memory only and lineage is discarded. feature_manifest (v0.4.0) maps column ↔ label ↔ definition, but to_tables() persists feature-group tables with no manifest table beside them; Feature.parents and stack_depth (the derivation chain) never reach the manifest; Feature.description defaults to "a feature" even though cli.py carries per-primitive documentation dicts (AGGREGATION_DOCS / TRANSFORMATION_DOCS).

Solution

Three coordinated changes, ordered so each ships green:

1. Key-name correctness. The aggs CTE reads the child table, so it must project and group by relationship.child_key; the existing join (cte.{child_key} = parent.{parent_key}) is then already correct. Propagate the same fix through the sharding metadata (key_columns, prefix/suffix) and the issue-#7 MaterializationKey. In the forward direction, _build_direct_cte projects relationship.parent_key from the parent's transform and joins cte.{parent_key} = child.{child_key}. Audit _build_direct_asof and the synth builder for the same implicit parent_key == child_key assumption.

2. Relationship identity. Relationships gain an optional config key name:. The planner derives a naming alias per relationship: the explicit name, else the child alias. Only-when-ambiguous rule: when one (parent, child) entity pair appears once — every config valid under v0.4.2 — the default alias equals today's alias, so every existing feature name, CTE name, Parquet column, and persisted feature-importance key is byte-identical (the ADR-0007 stability contract). When the pair repeats, validation errors unless every such relationship carries a distinct name: — loud, not silent. The alias flows into _build_name (via the relationship= kwarg aggregators already receive) and into both CTE-name templates.

3. Manifest persistence + lineage + descriptions. ManifestEntry gains depth, parents (immediate parent labels), relationship (naming alias, when aggregated), and a generated description templated from AGGREGATION_DOCS/TRANSFORMATION_DOCS. to_tables() additionally writes "<schema>"."<stem>_manifest" — one row per output column, including which feature-group table the column landed in — with the same idempotent DROP+CREATE and caller-owned-transaction contract as the group tables.

Notes

Decisions locked into this plan

DecisionChoiceRejected alternative & why
Naming default Only-when-ambiguous: unique (parent, child) pair keeps child.alias; repeated pair requires explicit name: Always-qualify (orders[buyer_id] everywhere) — renames every existing feature, breaking Parquet columns and persisted feature-importance keys (ADR-0007 contract). Auto-suffix only-on-collision — still silently renames leg 1 the day leg 2 is added; explicit names make the churn visible in the config diff.
Ambiguity handling Validation ERROR, not warning Warning + auto-dedupe — v0.4.2 already fails silently here; half-loud is still a silent wrong matrix for anyone not reading logs.
Bug-1 fix orientation CTE projects/groups by child_key (the column that exists on the table it reads); join untouched Aliasing child_key AS parent_key in the CTE — makes the join read naturally but re-introduces the name conflation the bug came from, and breaks when one child has two FKs to the same parent (exactly bug 2's topology).
Description storage Generated at manifest-build time; Feature.description untouched Populating Feature.description at construction — touches every primitive call site for zero additional information; manifest is the single consumer.
Version 0.5.0 (minor) 0.4.3 (patch) — new config key, new persisted table, and a formerly-accepted (silently broken) config now errors: that's minor-worthy surface change under the project's semver intent.

Risks & open threads

  • The collapse mechanism is partially unconfirmed. Empirically: 5 of 10 features, one CTE, one join. Whether the second relationship is skipped by the _path guard or processed-then-collapsed determines if the traversal loop needs surgery. Phase 2.1's trace test answers this before any naming code is written — do not skip it.
  • Name-stability is the contract. triage-pg pins featurizer by tag and consumes "<entity>.<col>=<value>" one-hot names (ADR-0007) plus feature-group tables keyed by column name. The Phase-2 snapshot test is the tripwire; if it ever fails, the design is wrong, not the test.
  • relationship.name enters CTE identifiers — it must satisfy PostgreSQL identifier rules after composition ({name}_aggs_for_{target} ≤ 63 bytes or routed through pg_identifier). Validation should reject names that can't survive composition rather than truncating silently.
  • SubqueryAggregator families (gap/entropy/gini/ngram/streak) embed the child alias and join key inside correlated subqueries — they are exposed to both bugs and are easy to miss; the Phase 1/2 audits explicitly include them.
  • Manifest table write uses INSERTs, not CREATE TABLE AS — definitions and labels contain single quotes and arbitrary SQL text; parameterized executemany, never string-formatted VALUES.
  • No new dependencies. Everything here is stdlib + existing deps; no uv add required.

Explicitly out of scope

  • Parent-entity direct categoricals (tracked separately in TODO.org since v0.4.1).
  • Upstream dssg PR decision (recurring deferred item).
  • Any change to one-hot naming or the imputation contract.
