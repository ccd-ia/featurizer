Feature-materialization performance — plan

Why the full-aggregator matrix is slow on real data, what actually causes it, and the ordered program to fix it. Grounded in live-DB measurement (dirtyduck / chicago311 / donorschoose), 2026-07-08.

1 · Symptom

After the v0.6.0 set-based rewrite + the companion-CTE cap fix, the all-aggregator config now materializes on every live DB — but slowly on the larger event streams:

DBrowsall-agg featuresall-agg seconds
dirtyduck22169272356.8
chicago3113065419110.1
donorschoose30001063281.1

chicago311 (more rows) is 35× faster than dirtyduck — so it is neither row count nor feature count alone. dirtyduck's config differs by using longer windows (P1M/P3M/P6M vs P7D/P30D/P90D) over 3 child categoricals (result/risk/type).

2 · Root cause (measured, not guessed)

CORRECTION (2026-07-09, superseding the fan-out analysis below). An EXPLAIN (ANALYZE) of the slow column-group — run late — shows the 343s is 9 correlated SubPlans over inspections_transform at loops=18909 (correlated subqueries re-executed once per target row), not the fan-out this section originally blamed. They are the two-window drift families kl_drift / wasserstein_drift — which ADR-0010 explicitly deferred as a non-goal but which fire on dirtyduck's 3 categorical columns × 3 intervals (2×3×3 = 18 correlated subqueries). Proof: dirtyduck all-agg MINUS those two families = 25.2s (vs 356.8s); the set-based fan-out is cheap (a 66-CTE group = 5s). The earlier claim here that "no correlated subqueries remain / the cost is pure fan-out" was wrong. The real fix is migrating the 2 drift families to set-based (the ADR-0010 pattern), or dropping them from the default-active set — far smaller than P2/P3. The analysis below is kept for the record but is superseded on this point. DONE (2026-07-09): both drift families migrated to set-based (ADR-0012) — dirtyduck all-agg 356.8s → 27.6s (~13×), all 272 features kept, values proven identical by the golden gate.

Each aggregator family was materialized on the full dirtyduck cohort in isolation. Every one is fast:

The buckets sum to ~17 s; the combined all-agg is 356.8 s — a ~20× super-linear blowup. So the cost is not a slow primitive. It is the monolithic single-query assembly: the full config emits ~38 CTEs (each set-based family × interval × categorical column is its own companion CTE — a window pre-pass over the child stream), all LEFT-JOINed into one synth CTE inside a lateral over as_of_dates. Aggregator coverage confirms it is not the correlated tier: 31 aggregators are plain GROUP BY, 27 are set-based companion CTEs (ADR-0010), and the 9 still-correlated ones are spatial/predicate/ drift families that do not fire on dirtyduck.

Two mechanisms produce the super-linearity, both fixable:

  • Planner (dominant). A ~38-way join exceeds PostgreSQL's default join_collapse_limit (8) and geqo_threshold (12), so join order is chosen by the genetic optimizer (GEQO) — effectively random — instead of cost-based search. A bad join order on wide intermediates is easily 10–100×. Crucially, you cannot just turn GEQO off: forcing exhaustive cost-based search over ~38 relations is super-exponential and killed the backend mid-plan (measured). The only durable escape is fewer relations per join — i.e. P2/P3.
  • Memory. Each companion CTE auto-materializes (referenced by the synth join), and the multi-way hash join of many wide intermediates spills to disk when work_mem (default 4 MB) is small.

2a · Does PostgreSQL tuning actually move it? (measured)

Full dirtyduck all-agg, same query, two connections — server defaults vs a tuned session (work_mem=256MB, join_collapse_limit=from_collapse_limit=30, geqo=off):

sessionsecondsvs baseline
baseline (server defaults, work_mem 4MB)359.31×
aggressive (work_mem 256MB, collapse 30, geqo OFF)backend killedplanning blowup
conservative (work_mem 64MB, collapse 20, geqo ON)259.91.38×

Verdict: tuning is a real but partial lever, and only when conservative. Modest work_mem (64 MB) buys ~1.4× by cutting hash-join spills — safe, cheap, worth shipping as an executor default. But it gets nowhere near the ~17 s the isolated buckets prove is achievable, because the dominant cost is GEQO picking a poor order for the ~38-way join, which more memory cannot fix. Worse, the naive "crank everything" (256 MB + geqo=off) was actively harmful: geqo=off forces exhaustive dynamic-programming join search over ~38 relations and the backend was killed mid-plan — and since the host is 128 GiB and the container is uncapped, this was planning explosion, not execution OOM. Conclusion: ship conservative tuning as a supporting default (P1), but the actual "must" is the structural fix (P2/P3) — bound joins-per-query so the count drops under geqo_threshold and cost-based planning returns, then run the resulting independent shards in parallel.

3 · The program (ordered)

P1 PostgreSQL planner/memory tuning as an executor default

The cheapest, highest-leverage lever, and the one to answer first because it is a config change, not an architecture change. featurizer's executor / to_dataframe / to_tables should issue a small set of SET LOCAL GUCs on its own connections before running the group queries: a modest work_mem bump (measured ~1.4× at 64 MB) and a small raise to join_collapse_limit / from_collapse_limit. Leave GEQO on — disabling it (or raising the threshold above the join count) makes exhaustive planning of a wide join explode and crashed the backend in testing. Constraints:

  • Only ever SET LOCAL (transaction-scoped) and only on connections featurizer opens itself. When the caller passes connection=, featurizer must not mutate their session — expose the recommended settings and let them opt in.
  • Values are advisory defaults with an override hook; work_mem is per-sort/hash-node, so it is bounded, not global.

P2 — REFUTED Parallelize + always-shard the assembly

Validated before building — the premise does not hold. The plan was: shard below 1664 columns to cap CTEs-per-query, then run the independent groups in parallel. Measured on dirtyduck all-agg:
testresultimplication
shard into 3 / 6 / 12 col-groups (serial)347.7 / 351.5 / 349.5stotal work CONSTANT — sharding doesn't reduce it
per-group timing (3 groups)343.0 / 5.0 / 2.2sone group holds it all → parallel wall-clock ≈ 343s
ANALYZE as_of_dates (fix no-stats estimate)376.7sthe 2550-row misestimate was not the runtime cost
Why: the EXPLAIN + string counts show the child event stream inspections_transform is scanned directly ~101 times across 105 CTEs (303 companion _for_ references) — each companion CTE an independent set-based pre-pass — all merge-joined into a ~100-way join. No correlated per-target-row subqueries remain (ADR-0010's migration is complete; the sub aliases in the plan are the pre-pass subqueries, not correlated scans). That fan-out is intrinsic to the plan shape: column-sharding just relocates the heavy columns into one group, and parallelism cannot split a single group's serial 100-way merge-join / scan chain. The tuning win (P1, 1.4×) is real join-ordering headroom on top, but the dominant cost is the fan-out, which only P3 addresses. P2 is therefore not implemented.
Is this dirtyduck-specific? No — it is general. The fan-out was measured DB-free across all three configs:

The fan-out is present in every config and grows multiplicatively with (families × columns × intervals); 2 of the 3 real DBs are already slow. chicago311 is fast only because it sits at the low end of every axis (fewest CTEs, smallest event stream, 7–90-day windows vs 1–6-month), not because it escapes the mechanism. Cost ≈ (# companion CTEs) × (child-stream size) × (join-width penalty) — and all three grow on a larger / richer DB, so this worsens with scale. It is a structural property of emitting the full aggregator set as one query, not a dirtyduck quirk.

Metadata under sharding (the question that motivated P2 — answered, still true even though P2 is shelved). Nothing breaks. The manifest is built once from the logical plan (plan.target_output_features), so it always covers every output column no matter how many data shards there are — sharding splits values, never metadata. There is exactly one <stem>_manifest table; each row carries a feature_group column pointing to the <stem>_group_NNN shard its values live in, and the short-name (63-byte cap) is computed identically on both sides so they join by column name. One hardening item survives regardless: _write_manifest_table defaults an unmatched column to group_000 — a silent mis-tag if the manifest and column_groups() ever disagree; add an assertion + test.

P3 (promoted to the primary fix) Reduce child-stream fan-out

The measurement above makes this the real lever, not an afterthought — and it is a single clean change (no correlated-subquery migration needed; ADR-0010 already did that). Today the child stream is scanned ~100× because a companion CTE is emitted per (family, column, interval), each independently scanning it. Share one interval-scoped pre-pass across families/columns — one ordered scan of the child stream feeding many reductions — instead of ~300 near-identical companion-CTE references. That cuts both the scan count and the ~100-way merge-join width, the two compounding factors. Architectural change to companion-CTE generation; needs its own spec and the golden-value equality gate (values/names preserved by construction, ADR-0007/0010).

P4 — DONE Guard ln / sqrt / log against out-of-domain inputs

Separate correctness bug surfaced by the wide variant: ln/log/ sqrt raised cannot take logarithm of a negative/zero on signed features (z-scores, diffs, deviations), aborting the whole matrix. Shipped Option A (NULL out of domain) as DomainGuardedTransformer — case when x > 0 then ln(x) end (>= 0 for sqrt) — recorded in ADR-0011. Names/labels unchanged (ADR-0007). Verified end-to-end: DB-free 574 passed (+3 guard tests); live-PG semantics probe (NULL for x≤0, sqrt(0)=0); and chicago311 wide — which crashed before — now materializes 30654 × 909 cols in 49.2s.

P5 Re-baseline ADR-0010 acceptance on real DBs

The ADR-0010 <120 s dirtyduck all-agg target was measured on synthetic 10k data and was never actually executable on the real DB (the config SyntaxError'd pre-fix). Re-state acceptance with real numbers after P1–P3 land: targets per DB, not one synthetic figure.

4 · Sequencing rationale (revised after measurement)

P4 — DONE (independent correctness fix, shipped). P2 — REFUTED and shelved: measured before building, sharding+parallelism cannot help this bottleneck (the cost is one group's 811× child-stream fan-out, not a distributable wide join). P3 is now the primary structural fix and the recommended next build: kill the 18 residual correlated subqueries, then share one child-stream pre-pass across families. P1 (conservative work_mem/collapse tuning, ~1.4×) is a cheap supporting default worth shipping alongside P3, GEQO left on. P5 re-baselines acceptance once P3 lands. The lesson: validate the premise on real data before building the infrastructure — the "obvious" parallel-sharding win evaporated under measurement.
