Purpose

Cut Featurizer 1.0.0 as a stability promise, not a feature release. The surface is already there (150 registry primitives, three planner passes, 18 φ-bridge families, sharding, manifest, categoricals — v0.9.1). What 1.0 adds is the right to rely on it: every claimed compatibility is tested, every known sharp edge is guarded or loudly documented, the 0.9.x families are validated at realistic scale, and the public surface is declared frozen under a written semver/deprecation policy (ADR-0015) that a human has actually reviewed.

Problem

The gap between what the project claims and what it verifies: requires-python >=3.10 is claimed but CI runs one Python on one PostgreSQL (16); the 0.9.x families passed synthetic integration tests but never the live 3-DB matrix that v0.6.0 and v0.8.0 earned their releases against; four known sharp edges sit open in TODO.org (a silent group_000 manifest fallback, the to_tables 8160-byte heap-row limit, the untested impute × materialized combination, the as-of-LATERAL materialization residual); ADR-0014 — the contract 18 families now stand on — was self-reviewed only; and there is no written statement of what is frozen, so "stable" would be a vibe, not a commitment.

Solution

Six phases, cheapest-first so every later phase runs on a hardened base: (1) close the four engineering guards; (2) widen CI into the claimed compatibility matrix; (3) re-run the live-DB revalidation matrix including new graph/text workloads and commit the artifacts (house convention: specs/live-db-revalidation-v100/); (4) write the API-freeze ADR-0015 + deprecation policy and put ADR-0014/0015 through an explicit human review gate; (5) ship docs-per-release and retire the stale org overview; (6) cut 1.0.0 through the existing tag-triggered pipeline. No new feature families anywhere in this plan — the taxonomy's remaining GAPs (peer churn, Moran's I, SBM, temporal motifs, LLM extraction) are explicitly post-1.0 lines.

[x] Phase 1: Close the engineering guards small, unblocks everything

The four known sharp edges from TODO.org. Each becomes either a hard guard with a test or a loud, documented boundary — nothing stays silent.

1. Manifest-under-sharding assertion

  • [x] Replace the column_to_group.get(entry.column, "group_000") fallback in _write_manifest_table (featurizer.py ~L675) with a fail-loud error naming the orphaned column and the available groups (a mis-tagged manifest silently corrupts lineage downstream — triage consumes this table).
  • [x] Integration test: build a sharded config, read back <stem>_manifest, assert every row's feature_group is a real persisted table and every output column appears exactly once.

2. to_tables heap-row-width guard

  • [x] Pre-flight in to_tables: estimate per-group row width (numeric = 8 bytes + header; text unbounded → skip estimate, document); when a group's fixed-width estimate exceeds ~8000 bytes, split that group further (reuse the column-group splitter with a lower cap) rather than let PostgreSQL fail with "row is too big".
  • [x] DB-free test pinning the estimator + the downshift decision; integration test with a ~1200-numeric-column group proving CTAS succeeds where it previously crashed.
  • [x] Document the limit + behaviour in the FAQ ("row is too big") and engineering/internals.md.

3. impute=True × materialized oversized-child path

  • [x] Integration test combining both: a config that triggers oversized-child temp-table materialization, executed via to_arrow(impute=True) and to_dataframe(impute=True); assert count-features → 0, measures keep NULL + __missing indicators, and values equal the unimputed run everywhere else.

4. As-of-LATERAL materialization residual — keep loud, teach it

  • [x] Keep the fail-loud NotImplementedError (sharding.py ~L1103) — implementing flat materialization of a correlated LATERAL is feature work, not hardening. Add/verify a test asserting the error fires with its actionable message.
  • [x] Document the boundary in the FAQ and the configuration reference's temporal-join section (when it bites, the two workarounds the message names).

Dependencies & environment

  • No new runtime dependencies anywhere in this plan. Possible dev-only addition: the starlight-versions npm package (Phase 5.2, best-effort).
  • Phase 3 needs the triage-side live-DB containers for chicago311; dirtyduck/donorschoose come from just seed. Port collisions with other projects' containers are a known hazard — use just --set pg_port <alt> (learned in the 0.9.x builds).
  • Phase 2's oldest-Python leg may surface syntax/typing drift (e.g. 3.10-incompatible unions in newer files) — fix forward, never raise the floor silently.

Risks

  • PG 14 leg fails on some SQL construct → decide floor honestly: fix if cheap, else declare "PG ≥ 16 tested, ≥ 14 best-effort" in README — but never claim untested support.
  • Live matrix regression vs v0.8.0 numbers → stop-the-line for Phase 6; the perf history (ADR-0010/0012/0013) shows regressions here are findable and fixable.
  • Human gate stalls → the plan is deliberately sequenced so Phases 1–3 are unaffected; only the tag waits on the signature.

What 1.0 explicitly is not

No new feature families, no engine changes, no PyPI, no GUI, no second framework. Anything that adds surface belongs to 1.1+ plans referencing the taxonomy GAPs and the deferred [bridge-graph] path.
