Purpose

Turn the current single-page GitHub Pages site into a connected documentation hub: a narrative walkthrough tutorial, the five example notebooks rendered as browsable pages, and generated reference pages (primitives, configuration, ADRs, changelog) — all single-sourced from files already in the repo, so the GitHub view and the site can never drift apart. Built on Astro Starlight, the same docs stack triage is adopting, so ccd-ia projects share one docs mental model, one theme, and one CI pattern. This directly answers: "the GitHub structure seems difficult to read and disconnected; Pages should contain a walkthrough tutorial and links to the notebooks."

Problem

The project's documentation is complete but scattered and mutually unaware. The README is a 500-line org-export that must be read top-to-bottom; the five executed tutorial notebooks are invisible unless you browse examples/; docs/ mixes org files (some only rendered as PDF), markdown references, and 13 ADRs with no index; the validation artifacts live under specs/; and the Pages site is a landing page whose links mostly bounce back to GitHub blobs. A new user has no guided path from "what is this?" to "I generated my first feature matrix", and no page answers "what primitives exist?" or "what goes in config.yaml?" without reading source.

Solution

An Astro Starlight site with a thin, testable Python pre-build seam. Authored content (walkthrough, configuration reference, landing/splash, notebooks learning path) lives as plain markdown in src/content/docs/ — still readable on GitHub. Generated content is produced by site/gen.py (uv toolchain, nbconvert only): the five notebooks convert from their committed outputs to markdown pages inside the theme (sidebar + Pagefind search included), the primitives reference is emitted from the live registry (with a count-parity test so it can never lie about the 67 + 83), and ADRs/CHANGELOG are ingested with injected frontmatter — their canonical homes stay docs/adr/ and CHANGELOG.md. The self-contained validation artifacts pass through public/ untouched: they are lab reports, not docs pages, and keep their own identity. Starlight's theme tokens carry the teal palette so featurizer and triage read as siblings. Generated files are gitignored — CI regenerates them on every deploy, so they cannot drift.

Questionables

Why Astro Starlight — and the decision history (ad-hoc → MkDocs → Starlight)

The plan originally chose a hand-rolled builder, then flipped to MkDocs Material on re-examination (two of three objections didn't survive scrutiny). The deciding fact arrived later: triage — the flagship consumer — is adopting Starlight, which converts Starlight's two biggest costs (Node toolchain in a Python shop, maintainer unfamiliarity) into sunk costs paid at the org level, and buys one docs mental model, shared teal theme tokens, cross-linkable sibling sites, and best-in-class search/UX. Starlight's real weakness for featurizer — no native notebook story — is closed with ~80 lines of nbconvert-to-markdown glue owned in the uv toolchain. featurizer doubles as the org's Starlight guinea pig: smaller site, learn here, harvest the config for triage.

Notebooks: convert committed outputs, never execute in CI

The Pages workflow has no PostgreSQL, and — more importantly — the committed outputs are the ones validated against a live database locally. Executing in CI would either fail (no DB) or silently show unvalidated results. Consequence to accept: contributors must re-execute notebooks locally when they change them (already the repo's practice; note 05 currently has uncommitted local edits that should land or be reverted before the site renders it).

Generated pages are gitignored, not committed

Everything gen.py emits (notebook pages, primitives reference, ADR/changelog copies) is regenerated on every CI build and gitignored locally. Committing generated content invites drift and noisy diffs; the cost is that a local npm run dev needs uv run python site/gen.py first — documented in the README docs block and cheap.

Walkthrough as markdown, not a sixth notebook

Markdown is readable on GitHub, diffs cleanly, renders in the theme natively, and can interleave shell + Python + YAML naturally (a notebook privileges Python cells). The notebooks remain the deep-dive layer; the walkthrough is the narrative spine that links into them. Its truthfulness is enforced by executing every command during the build phase. (Quarto's freeze-cached executable walkthrough was the one capability we give up in the Starlight decision — recorded as a trade-off, revisit only if the walkthrough proves hard to keep honest by hand.)

What other documents should we include? — recommended vs deferred

Now (this plan): walkthrough, rendered notebooks + learning path, generated primitives reference, configuration reference, ADR index, changelog. Deferred (listed in Notes): auto-generated API reference (pdoc → public/api/ pass-through), an interactive primitives explorer (Astro component — the payoff Starlight uniquely enables), FAQ/troubleshooting, the org-mode deep-dive and overview docs, and a performance-methodology page (the validation reports already carry the numbers). Deferring keeps this plan shippable in one pass; each deferred item is additive later.

Node maintenance tax: accepted, org-level

package.json/lockfile bumps and Dependabot noise for Astro/Starlight are accepted as an org-level cost shared with triage (same stack, same upgrade rhythm). Pin engines.node >= 20, use npm ci in CI for reproducibility, and batch dependency bumps rather than merging each bot PR.

docs/primitives-reference.md: keep or retire?

Keep for now with a "canonical view is the generated page" pointer — it is linked from existing commits/PRs and costs nothing. Retire in a later housekeeping pass if it drifts. The generated page wins because it cannot disagree with the registry (the count parity test enforces this).

Dependencies & constraints

  • Python side: docs dependency group = nbconvert only (uv-locked). Runtime package untouched; gen.py imports the registry read-only.
  • Node side: Astro + Starlight in package.json/package-lock.json; Node ≥ 20; npm ci in CI. First Node footprint in the repo — accepted org-level (see Questionables).
  • Base path: GitHub project pages serve at /featurizer/; astro.config.mjs must set site + base, and all internal links/assets must survive it (Phase 1 spike #1).
  • The Pages workflow gains ~2–3 min (uv sync + gen + npm ci + astro build). Concurrency group already prevents deploy races.
  • The current working tree has in-flight code changes (planner/abstractions/tests, notebook 05). This plan touches none of those files; notebook 05 renders only from its committed state.
  • Theme tokens (--sl-color-accent* teal set) should be written to be copy-pasteable into triage's Starlight site — one palette, two repos, until an org-shared theme package is worth extracting.

Rejected approaches (decision log)

  • Ad-hoc site/build.py (this plan's v1) — bespoke ~300 lines nobody else recognizes; no search; every feature hand-rolled. Rejected once org standardization entered the picture.
  • MkDocs Material (this plan's v2 recommendation) — excellent pure-pip fit (mkdocs-jupyter, mkdocstrings, --strict), but it would make featurizer the org's odd one out once triage runs Starlight. Superseded by alignment, not by defect.
  • Quarto — best native notebook support and the freeze-cached executable walkthrough was genuinely attractive; loses on org alignment and the second non-uv toolchain. The freeze idea is recorded in Questionables as the capability we consciously gave up.
  • Linking notebooks to GitHub's ipynb viewer only — keeps the "disconnected" feel, no nav/search/identity, truncates large outputs.
  • Executing notebooks in CI — no database in the Pages workflow; committed outputs are the validated ones.

Risks & mitigations

  • Base-path 404s — Phase 1 spike #1; check_links.py over dist/ in CI thereafter.
  • DataFrame HTML tables render poorly in the theme — Phase 1 spike #2 against notebook 05 (the worst case) before any content investment; the loop instruction halts tooling on structural failure.
  • Docs drift — primitives page generated (cannot drift; count-parity test fails CI when the registry moves); ADR/changelog ingested from canonical homes; generated pages gitignored so there is no second copy to rot.
  • Node dependency churn — batch bumps, lockfile + npm ci, org-shared upgrade rhythm with triage.

Future work (explicitly out of scope)

  • Interactive primitives explorer (Astro component, faceted filter + SQL preview); pdoc API reference under public/api/; org-shared Starlight theme package; versioned docs per release tag; org→HTML export of the deep-dive.
