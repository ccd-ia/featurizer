Purpose

Implement the missing text (substrate E) and graph (substrate C) feature families described in docs/text-network-features/temporal-features-network-text.org, delivering them as real, tested φ-bridge computers (not just documented interfaces) while preserving the point-in-time causal boundary (ADR-0001), keeping heavy dependencies behind the [bridge] extra, and requiring no change to the SQL engine — the spine keeps aggregating materialized columns exactly as today. This is the 0.9.0 feature line.

Problem

The taxonomy doc is a rich roadmap, but the code ships only four exemplars — one per modality (PageRankBridge, TfidfTopicShareBridge, SentenceEmbeddingBridge, MarkovSurprisalBridge). Everything else — graph centralities / community / SBM / temporal motifs, text NER / sentiment / toxicity / readability / LLM-extraction, embedding drift / novelty — exists only as an abstract-interface list in featurizer/bridge/__init__.py:27-47. Worse, two structural limits in the current BridgeComputer contract block whole families before we even start:

  • Single φ per bridge. compute() → {pk: scalar|vector}. Multi-metric families (NER counts per entity type; a node's degree + betweenness + coreness from one graph build) would need one bridge — and one full pass — per metric.
  • Single graph snapshot. PageRankBridge fits one graph at one fit_before. But centrality is non-local — one future edge changes it — so a cohort with many as-of dates needs the graph rebuilt per window. There is no snapshot-sequence mechanism.

Solution

Do not touch the SQL engine and do not write a second feature framework. Instead: (1) extend the BridgeComputer contract minimally to remove the two structural limits, then (2) fill the high-value families per modality by subclassing it, exactly following the four exemplars. Every new family fits the same causal-guarded compute → materialize → emit_yaml lifecycle, so the spine consumes the results unchanged. Heavy deps stay behind [bridge] (and one very heavy path behind a separate extra). The contract extension is a hard-to-reverse public decision, so it earns ADR-0014.

[] Phase 1: Bridge harness extensions (the enablers)

Remove the two structural limits before adding families. All additive — the current single-column, single-snapshot path keeps working. This is the phase that earns ADR-0014.

  • [] Add a MultiColumnBridge subclass (or a value_cols: list[str] mode on the base) whose compute() returns {pk: {col: val}}; keep the scalar/vector base untouched for backward compat.
  • [] Generalize materialize() to build the DDL and INSERT for N value columns (reuse _carry_type / _column_type per column).
  • [] Generalize emit_yaml() to declare one Variable per value column with correct per-column types.

2. Temporal snapshot-sequence helper

  • [] Add a helper that, given a set of as-of dates, rebuilds the graph/model on the pre-t₀ slice per window and keys the output by (entity, as_of_date) — the causal-correct "snapshot sequence" representation from the doc §C②.
  • [] Emit the snapshot output as an event stream (carry as_of_date as the temporal_ix) so the existing spine aggregates "trend in centrality" as a normal window over a metric.
  • [] Assert the boundary per window via assert_pre_t0; document the O(windows × build) cost in the docstring (see Questionables).

3. Persistent materialization option decided: persist, temp default

  • [] Add a persist: bool (or on_commit) parameter to materialize(): create temp … on commit drop (default, today's behavior) vs a persistent create table for real Dagster/Snakemake assets (ADR-0003 wiring).

[] Phase 3: Graph substrate — centralities & community care decided: cheap default, heavy opt-in · defer SBM

[] Phase 4: Native cheap-graph tier — 1-hop neighbour aggregation (SQL planner pass) decided: build in 0.9.0

The doc's cheapest graph tier (§C③ "neighbour aggregation" + the cheap-tier SQL example), built in-spine — no Python, no [bridge] dependency. This is a deliberate engine change (the one exception to "no planner changes"): a config-level graph/edge concept plus a generated recursive/1-hop SQL pass, modeled on the existing peer_groups / spatial_relationships planner passes (not a registry primitive).

1. Config surface + validation

  • [] A graph_relationships (or event_edges) config block declaring the edge source (src, dst, ts columns) and the neighbour property to aggregate — parallel to the peer_groups / spatial_relationships blocks.
  • [] Extend featurizer/validation.py to validate the block (columns exist, types, temporal column present) with the same error/typo-suggestion quality as the rest.

2. Planner pass (as-of 1-hop SQL)

  • [] Generate the as-of aggregate in featurizer/planner.py: degree_<window> and 1-hop neighbour mean/share, bounded by e.ts < aod.as_of_date and the neighbour state valid_at < as_of_date (the doc's cheap-tier SQL is the template). Emit as feature columns keyed by (entity, as_of_date) — a planner pass, not a registry primitive.
  • [] 1-hop only for 0.9.0: explicitly do not generate 2-hop aggregates (the canonical temporal-GNN leakage — 2-hop pulls neighbours' future labels). Enforce/document the boundary.

3. Testing Strategy

Questionables

All resolved 2026-07-16 (via AskUserQuestion). Each decision is folded into the phases above; kept here as the decision log.

✓ Decided — Multi-column output: extend the contract (verify no regression)

MultiColumnBridge: compute → {pk: {col: val}}, one pass emits many columns. The base single-column/vector path stays for backward compat — purely additive, and a mandatory regression test proves the existing path is byte-identical (Phase 1, task 5). This is the ADR-0014 decision.

✓ Decided — Temporal graph cost: cheap centralities default, ALL heavier metrics opt-in

Non-locality forces a per-window rebuild. Default to the cheap tier (degree, coreness, clustering); every heavier metric — betweenness, eigenvector, closeness — is opt-in behind a flag (not just betweenness). No snapshot-binning approximation. Document the O(windows × build) cost knob so features never get silently 100× slower (Phase 3, task 1).

✓ Decided — SBM: defer; ship Louvain/Leiden (pip) now

graph-tool is not pip-installable and would break pip install 'featurizer[bridge]'. SBM / MDL-surprise (the coordination engine) lands later behind a separate documented [bridge-graph] path; Louvain/Leiden cover community detection in 0.9.0 (Phase 3, task 2).

✓ Decided — Cheap 1-hop neighbour aggregates: build the native SQL planner pass in 0.9.0

Changed from the initial "defer to 0.10" recommendation. The cheapest, highest-value graph tier (degree, 1-hop neighbour mean/share) is built in-spine now — no Python, no [bridge] dep — as a new graph_relationships config block + generated as-of SQL, modeled on peer_groups/spatial_relationships. This is the one deliberate engine change, now its own Phase 4. 1-hop only (2-hop leaks neighbours' future labels).

✓ Decided — Pretrained-model vintage: record metadata + document

assert_pre_t0 guards fitted models only; a pretrained snapshot trained on post-t₀ data is silent leakage. Add an optional model_vintage attribute to BridgeComputer, pin model versions, surface the training cutoff in metadata so a strict backtest can assert vintage ≤ as_of, and document the caveat in ADR-0014 + the cookbook (Phase 1, task 4).

✓ Decided — NLP language: multilingual defaults, per-bridge language=

The working domains are Spanish. Default to multilingual/es models (spaCy xx_/es_core_news, a multilingual sentence-transformer, a Spanish sentiment lexicon); expose a per-bridge language= parameter so English/other is a one-line override (Phase 2).

✓ Decided — Materialization: add persist option, temp default

Keep create temp … on commit drop as today's default (reference flow); add a persistent create table for real Dagster/Snakemake assets (Phase 1, task 3; ADR-0003 wiring).

Invariants this plan must not break

  • No second feature framework. Every bridge family ends as a materialized column the existing spine aggregates — bridges need zero engine change. The one deliberate engine addition is Phase 4's native 1-hop neighbour pass (a new graph_relationships block + generated SQL, in 0.9.0 by decision), built like the existing peer_groups/spatial_relationships passes.
  • Causal boundary (ADR-0001). Every fitted model asserts assert_pre_t0; the snapshot loop re-asserts per window. Pretrained-model vintage is the one gap — documented, not silently ignored.
  • Optional deps stay optional (ADR-0003). The SQL spine never imports bridge deps; each bridge guards its import with the install-hint pattern. uv sync (no extra) must still type-check and run the fast suite.
  • Categorical outputs (community membership) flow through the existing ADR-0007 fixed-vocabulary one-hot path — no new categorical machinery.

Rejected / deferred approaches

  • One bridge per metric — rejected: wasteful re-computation; multi-column is strictly better.
  • SBM / MDL-surprise (graph-tool) — deferred (decided): not pip-installable; lands later behind a separate [bridge-graph] path. Louvain/Leiden (pip) cover community detection in 0.9.0.
  • 2-hop neighbour aggregates & temporal-GNN — deferred: 2-hop pulls neighbours' future labels (the canonical leakage); the native pass is 1-hop-only for 0.9.0.
  • LLM structured extraction (Path 3) — out of scope here: once extracted, the fields are ordinary columns (back to Path 1), but the extraction harness (schema, model, cost) is its own effort.

Recommended sequencing

Phase 1 is the keystone — do it first and get ADR-0014 reviewed before building families on the new contract. Phases 2 (text bridges), 3 (graph bridges), and 4 (native SQL pass) are largely independent and can be built in parallel once Phase 1 lands — Phase 4 is engine-side and doesn't depend on the bridge families. Phase 5 (trajectory/sequence/edges) depends on 2 & 3 (trajectory needs embeddings; text-induced edges feed the graph bridges). Cut 0.9.0 after Phase 4 (via Phase 6's release steps); Phase 5 ships as 0.9.1.
