Plan: Correlated-Subquery Aggregator Scaling

Remove the full-cohort scaling cliff of featurizer's advanced aggregator tier by replacing per-group correlated subqueries with set-based window pre-passes reduced by plain GROUP BY — measured first, migrated family-by-family, gated at every step by the full-registry execution harness and golden-value equality.

Purpose

Discharge the deferred remediation of ADR-0009: make the ~26 subquery-backed advanced aggregators (gap statistics, burstiness, entropy/hhi/gini, deviation statistics, the n-gram/sequence/Markov family) practical for full-cohort materialization. The acceptance bar is concrete: the all-agg dirtyduck materialization that today exceeds a 300 s budget over 22,169 target rows must complete in well under a minute, with byte-identical output column names (ADR-0007 / triage-pg contract) and value-identical results (proven against golden values captured from v0.5.2).

Problem

Every SubqueryAggregator emits a scalar correlated subquery into the aggs CTE (planner.py:1313): select child_key, (SELECT … FROM <child>_transform sub WHERE sub.child_key = <child>_transform.child_key …) … group by child_key. PostgreSQL evaluates that subquery once per group — and because <child>_transform is a CTE, which cannot carry an index, every evaluation is a sequential scan of the entire child stream. Total cost is O(as_of_dates × groups × subqueries × child_rows), where "subqueries" multiplies again per configured interval. Measured (ADR-0009): the narrow default tier materializes dirtyduck's 22,169 targets in 3.6 s; the all-agg tier times out at >300 s. This is purely a performance property — v0.5.2 proved all generated SQL is valid and correct.

Solution

Convert each aggregator family from G-per-group rescans into one companion CTE: an inner window pre-pass (e.g. LAG(ts) OVER (PARTITION BY child_key ORDER BY ts)) that scans the child stream once, wrapped by a plain GROUP BY child_key reduction that computes every family member's aggregate in the same pass. The companion CTE joins into the target's synth exactly like today's aggs CTE (same join, synth-column-source, sharding, and materialization machinery). Cost per family drops to one O(N log N) sort + one O(N) reduction — the same class as the default tier that already scales. The migration is opt-in per aggregator (a preagg protocol on SubqueryAggregator), so special-config families (predicate-driven, drift, spatial) keep the correlated path untouched until/unless they need it. Work is measure-first: Phase 0 builds the scaling benchmark and captures golden values, Phase 1 spikes the two ADR-0009 candidates head-to-head before any planner code changes.

Before / after, gap family

-- TODAY (per group, ×G, ×interval): correlated re-scan inside the aggs CTE
select c_transform.pid,
  (SELECT AVG(gap) FROM (
     SELECT <epoch_day_span>(sub.ts, LAG(sub.ts) OVER (ORDER BY sub.ts)) as gap
     FROM c_transform sub
     WHERE sub.pid = c_transform.pid AND sub.ts::date <= aod.as_of_date
   ) gaps WHERE gap IS NOT NULL) as "GAP_MEAN(c.ts)",
  … 5 more gap-family subqueries, each re-scanning c_transform …
from c_transform where ts::date <= aod.as_of_date group by pid

-- TARGET (once, whole family): companion CTE, window pre-pass + plain reduction
c_gaps_aggs_for_p as (
  select pid,
    avg(gap)                                                as "GAP_MEAN(c.ts)",
    stddev(gap)                                             as "GAP_STDDEV(c.ts)",
    min(gap)                                                as "GAP_MIN(c.ts)",
    max(gap)                                                as "GAP_MAX(c.ts)",
    stddev(gap) / nullif(avg(gap), 0)                       as "GAP_CV(c.ts)",
    (stddev(gap) - avg(gap)) / nullif(stddev(gap) + avg(gap), 0) as "BURSTINESS(c.ts)"
  from (
    select pid,
           <epoch_day_span>(ts, lag(ts) over (partition by pid order by ts)) as gap
    from c_transform
    where ts::date <= aod.as_of_date          -- (+ daterange window per interval variant)
  ) gaps
  where gap is not null
  group by pid
)
-- joined into p's synth on pid, exactly like c_aggs_for_p

Output column names come from the unchanged _build_name/_build_label — byte-identical to today. Only the definition and the CTE it renders in change. Keeping pre-pass + reduction as one CTE (subquery in FROM, not a chained CTE) means the existing ShardableCTE prefix/suffix machinery, synth pruning, and MaterializationKey registration apply without modification.

Design constraints that must hold throughout

  • Naming contract (ADR-0007 / triage-pg): output column names and manifest labels are produced by _build_name/_build_label, which this plan never touches. The name-stability snapshot in tests/test_parallel_relationships.py is the mechanical enforcement — it runs in every batch gate.
  • Interval semantics: the current subqueries filter the child stream to the interval window before LAG, so the first in-window event has a NULL gap. Companion CTEs replicate this by emitting one CTE per (family, interval) whose pre-pass applies the same daterange_window/causal filter before the window function. Never compute LAG globally and filter after — that changes boundary values.
  • Tie-ordering: order by ts with equal timestamps is nondeterministic today (correlated form) and stays so (window form). Gap values are tie-invariant (multiset preserved); sequence transitions are not — the golden comparison must treat tie-affected fixtures with set-level equality or use tie-free fixtures (Phase 0 fixtures are tie-free by construction).
  • NULL semantics: empty group → correlated subquery returns NULL; companion CTE has no row and the LEFT JOIN yields NULL — same observable value. Groups whose pre-pass rows are all filtered (e.g. single event → NULL gap) produce either no row or a NULL aggregate; both land as NULL. Golden equality covers both via the pid=1/pid=6 fixture groups.
  • One CTE per family, not two: pre-pass lives as a subquery in the reduction's FROM, so ShardableCTE prefix/suffix, synth pruning, and MaterializationKey registration work without touching sharding.py. A chained pre-pass CTE would need a CTE-dependency notion sharding doesn't have.
  • Per-aod recomputation is fine: companion CTEs sit inside the lateral like every other CTE; the narrow tier proves per-aod GROUP-BY recomputation scales (3.6 s / 22k targets). The cliff was G×S rescans, not the lateral.

Rejected / non-selected approaches

  • Pure LATERAL rewrite (select … from (select distinct key …) cross join lateral (…)): same asymptotics as today against an unindexed CTE — reshapes the query without removing the per-group rescan. Only reconsidered if the Phase 1 spike surprises.
  • Indexing the CTE: impossible in PostgreSQL; that is precisely why Option B materializes a temp table first. Option B remains the spike's control arm and the documented fallback if Option A's planner work stalls — but it keeps O(G×S) index probes and inherits ADR-0006's as-of materialization residual (NotImplementedError on as-of LATERAL in a materialized synth).
  • Status quo (bounded cohorts forever): ADR-0009's own framing marks this as the interim state, not the destination — the advanced tier exists for triage-style full-cohort matrices.

2026-07-06T20:20 — CTE-index premise empirically confirmed on PostgreSQL 16.14 (pre-build verification)

Before building, the load-bearing claim — "you cannot index a CTE, so the correlated subquery is forced to seq-scan the child stream per group" — was verified with an EXPLAIN (ANALYZE, BUFFERS) battery on the ephemeral postgres:16 (50k child rows, 500 groups, one avg subquery):

temp table + index (Option B)yes — Bitmap Index Scan19 ms

NOT MATERIALIZED + computed window columnpartial — index for the pid filter, but recomputes the window per group37 ms

Confirmed, with a refinement. The CREATE INDEX on a CTE is a hard error, but the deeper reason featurizer can't recover an index is twofold: (1) its _transform CTEs are referenced multiple times (the outer projection and each correlated subquery), so PostgreSQL auto-materializes them into an unindexed working table; and (2) even if forced inline with NOT MATERIALIZED, the transform layer carries computed/window columns, so the subquery still recomputes per group (37 ms case) rather than probing a usable index. This means a "just add NOT MATERIALIZED" shortcut does not rescue the real transform-layer case — it only helps a trivial passthrough. Early Phase-1 signal: Option A (set-based, 5.5 ms) beats Option B (temp+index, 19 ms) by ~3.5× and the status quo by ~100× on a single subquery; the multiplier compounds with 26 subqueries × intervals. This pre-answers the decision gate in Option A's favor, but the Phase 1 spike still runs the full gap-family head-to-head at 100/1k/10k before it is ratified in ADR-0010.
